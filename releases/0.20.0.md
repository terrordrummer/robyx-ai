## Unified Scheduler + Continuous Tasks

### Breaking: Three schedulers merged into one

The periodic scheduler (`bot/scheduler.py`, every 10 min), timed scheduler
(`bot/timed_scheduler.py`, every 60s), and reminder engine (`bot/reminders.py`,
every 60s) have been merged into a **single unified scheduler** that runs every
60 seconds and reads from `data/queue.json`.

**What changed:**
- `bot/timed_scheduler.py` — **deleted** (merged into `scheduler.py`)
- `bot/reminders.py` — **deleted** (merged into `scheduler.py`)
- `data/tasks.md` — no longer used by the scheduler (migrated to `queue.json`)
- `data/timed_queue.json` — no longer used (migrated to `queue.json`)
- `data/reminders.json` — no longer used (migrated to `queue.json`)
- `SCHEDULER_INTERVAL` default changed from `600` to `60`
- `TIMED_SCHEDULER_INTERVAL` env var — **removed**

**Migration is automatic.** On first boot after upgrade, the scheduler reads
all three legacy files, merges pending entries into `data/queue.json`, and
renames the old files to `.migrated` backups. No manual action needed.

### New: Continuous Tasks

A new task type for autonomous, iterative work. The agent executes one step
at a time, commits results, plans the next step, and terminates. The scheduler
picks up the next step on the following 60s cycle. This continues until the
objective is reached or the user intervenes.

**Features:**
- Dedicated workspace topic per continuous task (auto-created with 🔄 prefix)
- Git branch in the target project's repo (`continuous/<name>`)
  - Existing repo → creates branch
  - No repo → `git init` + branch
  - No git → proceeds without versioning
- State file at `data/continuous/<name>/state.json` tracks progress
- Step-by-step execution with commit per step
- User interruption: send a message in the continuous task's topic to
  interrupt the running agent (SIGTERM + 5s grace + SIGKILL)
- Rate limit handling: sets `rate-limited` status, scheduler retries hourly
- Interview/setup flow to clarify objectives before starting

**New files:**
- `bot/continuous.py` — state management (create, step lifecycle, pause/resume)
- `templates/CONTINUOUS_SETUP.md` — interview template for program definition
- `templates/CONTINUOUS_STEP.md` — step execution template

**Creating a continuous task** (from orchestrator or workspace agent):
```
[CREATE_CONTINUOUS name="<slug>" work_dir="<path>"]
[CONTINUOUS_PROGRAM]
{
  "objective": "...",
  "success_criteria": ["..."],
  "constraints": ["..."],
  "checkpoint_policy": "on-demand",
  "context": "...",
  "first_step": {"number": 1, "description": "..."}
}
[/CONTINUOUS_PROGRAM]
```

### Agent Interruption

Agents now track their running subprocess. When a user sends a message to a
continuous task's workspace while a step is executing, the subprocess is
interrupted (SIGTERM → 5s grace → SIGKILL) and the user's message is processed
immediately instead of queuing behind the lock.

### Queue entry types

The unified `data/queue.json` supports four entry types:

| Type | Purpose |
|------|---------|
| `reminder` | Plain text delivery at a precise time (no LLM) |
| `one-shot` | Agent subprocess at a specific date/time |
| `periodic` | Recurring agent subprocess on an interval |
| `continuous` | Iterative autonomous work, step-by-step |

### Compatibility

- **Auto-migration**: `tasks.md`, `timed_queue.json`, and `reminders.json`
  are migrated to `queue.json` at boot. Old files renamed to `.migrated`.
- **`[REMIND ...]` pattern**: unchanged. Works identically for all agents.
- **`scheduler.add_task(...)`**: replaces `timed_scheduler.add_task(...)`.
  Same arguments, new import path.
- **Workspace creation**: `[CREATE_WORKSPACE ...]` pattern unchanged.
- **`.env`**: remove `TIMED_SCHEDULER_INTERVAL` if present (ignored).
  Update `SCHEDULER_INTERVAL` to `60` if it was `600`.
