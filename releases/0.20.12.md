## Version-chained migration framework

Robyx used to track migrations by free-form string id (e.g.
`0.15.0-reset-sessions-for-reminder-skill`) through a single decorator
registry. That worked for one-off data fixes but had no defence against
multi-version jumps: if an install skipped from 0.18 to 0.21, only the
latest release's migration ran, and any intermediate schema change was
silently skipped.

This release introduces a **version-chained migration framework**. Every
release from 0.20.12 onward ships a matching `bot/migrations/vX_Y_Z.py`
module declaring `from_version` → `to_version` and an async `upgrade()`.
A CI-style contract test fails the build if any release is missing its
migration — even a release with no data changes ships a no-op, so the
chain stays continuous.

### How upgrades actually work now

On boot, `run_pending(platform, manager)` executes two layers back-to-back:

1. **Legacy name-keyed registry** (unchanged from 0.20.11) — idempotent
   one-off migrations from the pre-framework era. Kept for backwards
   compatibility with existing installs' `data/migrations.json` entries.
2. **Version chain** — the runner discovers every `vX_Y_Z.py` under
   `bot/migrations/`, validates that `to_version` of each step equals
   `from_version` of the next (no gaps, no forks, no duplicates), and
   runs every step whose `to_version` is greater than the tracker's
   `current_version`. On the first failure, the chain halts and the
   remaining steps stay pending for the next boot.

Multi-version jumps now run every intermediate step in order: 0.18 → 0.25
will execute the migrations for 0.19, 0.20, 0.21, 0.22, 0.23, 0.24, and
0.25, in that sequence. Nothing is ever skipped.

### Tracker format

`data/migrations.json` now holds both layers in the same file:

```json
{
  "0.15.0-reset-sessions-for-reminder-skill": { ... legacy ... },
  "0.19.0-rename-kaelops-to-robyx": { ... legacy ... },
  "_chain_": {
    "current_version": "0.20.12",
    "history": [
      {"from": "0.20.11", "to": "0.20.12", "status": "ok", "applied_at": "..."}
    ]
  }
}
```

On the first boot of 0.20.12, installs without a `_chain_` key are
seeded at `current_version = "0.20.11"` so the bootstrap migration
below runs next.

### The bootstrap migration

`bot/migrations/v0_20_12.py` is a no-op — the framework's arrival is
the only change this release carries. Its purpose is to advance the
tracker from 0.20.11 to 0.20.12 so future versions can chain off it.

### Scaffolding new migrations

`scripts/new_migration.py` creates a new `vX_Y_Z.py` wired to the chain:

```bash
python scripts/new_migration.py 0.20.13 --description "Rename specialist memory dir"
```

The scaffolder auto-detects the previous version from `releases/` and
emits a no-op `upgrade()` with the correct chain links. The runner
auto-discovers the new module via `pkgutil.iter_modules`; no manual
import edit needed.

### No migration needed at the user level

This release *is* the migration: it introduces the framework and
bootstraps every install to `current_version = 0.20.12`. Nothing in
`data/`, `.env`, `state.json`, `queue.json`, `continuous/`, or any
platform's channel / topic / session ids is touched. Telegram / Slack /
Discord IDs, scheduled tasks, and agent sessions all survive untouched.
