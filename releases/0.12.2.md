## Fixes

### Auto-update now reliably reinstalls Python dependencies

**Root cause**: when v0.12.0 added Pillow as a new dependency, some deployments booted against a venv that did not have Pillow installed, crashing `SEND_IMAGE` with `No module named 'PIL'`. The existing `apply_update` code *did* call `pip install -r bot/requirements.txt` at step 5 — but (a) it used `-q`, (b) it did not check the return code, (c) it captured stdout/stderr and threw them away, (d) it had a 120 s timeout. A failed install was therefore indistinguishable from a successful one: the update was reported as success, the service restarted, and the new code booted against a stale venv.

### What changed

**`bot/updater.py::apply_update` — robust pip install**

- Verbose pip output (removed `-q`).
- stdout and stderr are now logged at INFO level so the install trace is always in `bot.log`.
- **Return code is checked.** A non-zero exit rolls back to the previous version tag (`git checkout v{current}`) and fails the update with a clear error containing the last ~8 lines of pip output. The user sees the actual pip error in the chat, not a mystery success followed by a boot crash.
- Timeout bumped from **120 s → 600 s** to accommodate wheel builds on ARM / slow hardware.
- Preflight check: if `.venv/bin/pip` does not exist, fail the update immediately with a clear error instead of letting `create_subprocess_exec` crash later.
- On success, the bootstrap hash marker (see below) is refreshed so the next boot does not redundantly re-run pip for the same `requirements.txt`.

**`bot/_bootstrap.py` — new startup dependency safety net**

A new tiny module that runs at the top of `bot/bot.py`, before any other `bot/*` import. It:

1. Hashes `bot/requirements.txt` with SHA1.
2. Compares the hash against `.venv/.kaelops_deps_hash`.
3. If they match, returns immediately (common path — fast).
4. If they differ (or the marker does not exist), runs `pip install -r bot/requirements.txt` inside the bot's venv, logs the outcome to stderr, and writes the new hash on success.

This covers three scenarios the updater alone cannot handle:
- The user pulled the repo manually without running `pip install`.
- A previous auto-update finished its `git pull` but crashed before the pip step.
- The venv has been manually touched / corrupted between restarts.

The check is **per-venv** (the hash marker lives inside the venv itself), so different environments don't share state. It only uses the Python stdlib, so it can run even when some third-party dep is genuinely broken. A pip failure is logged loudly on stderr but not fatal — the bot still tries to import, because the eventual `ImportError` traceback is often more informative than a generic "pip returned 1".

### Tests

Suite grew from 621 to **632 tests** (+11). New coverage:

- `tests/test_bootstrap.py` (new, 7 tests): first-run installs and writes marker, same-requirements is a no-op, hash mismatch triggers reinstall, pip failure does not update marker, pip timeout does not crash, missing requirements is a no-op, missing venv is a no-op.
- `tests/test_updater.py` (+4 tests): `test_pip_nonzero_rolls_back`, `test_pip_timeout_rolls_back`, `test_missing_pip_binary_fails_cleanly`, `test_successful_pip_refreshes_bootstrap_marker`.

## Operator Notes

### If you are currently stuck on v0.12.0 / v0.12.1 with a broken venv

Manual one-shot recovery:

```bash
cd <your KaelOps install dir>
./.venv/bin/pip install -r bot/requirements.txt
```

That installs whatever is currently missing (Pillow on v0.12.0+). The running bot process does NOT need a restart — Python does not cache failed imports, so the next `SEND_IMAGE` will pick up the newly-installed module. The next auto-update to v0.12.2 will then refresh the bootstrap marker, and subsequent releases with new deps will install automatically.

### How to inspect the bootstrap marker

```bash
cat .venv/.kaelops_deps_hash    # SHA1 of the last requirements.txt pip-installed against this venv
sha1sum bot/requirements.txt    # what should be there
```

To force `_bootstrap.py` to re-run pip install on the next boot, delete the marker:

```bash
rm .venv/.kaelops_deps_hash
```
