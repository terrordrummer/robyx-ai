## Auto-invalidation of stale agent sessions on every update

A focused follow-up to v0.15.0 that promotes session invalidation from a one-shot migration into a permanent contract of `bot/updater.py`. After this release, **any future change to a system prompt or to an agent/specialist brief is automatically delivered to the affected agents on the very next boot after the update** — no per-release migration required, no chance of the v0.14 → v0.15 silent regression happening again.

---

### The contract

When `bot/updater.py:apply_update` finishes a successful fast-forward pull:

1. It captures the pre-pull HEAD with `git rev-parse HEAD` (done before the pull, so the SHA is the *previous* commit).
2. After the pull, deps install, and stash pop, it computes `git diff --name-only <pre_pull_sha> HEAD`.
3. It hands the changed paths to a new `bot/session_lifecycle.py:invalidate_sessions_for_paths(state_file, changed_paths)` helper.
4. The helper decides which agents need a fresh AI-CLI session based on the diff, mutates `data/state.json` in place, and persists.
5. The next interactive turn for each affected agent creates a brand-new Claude Code session that bakes in the up-to-date system prompt.

### The reset granularity

| Diff touches… | Effect |
|---|---|
| `bot/config.py` (the system prompts) | Reset **all** known agents |
| `bot/ai_invoke.py` (the per-agent brief loader) | Reset **all** known agents |
| `agents/<name>.md` | Reset only **that** workspace agent |
| `specialists/<name>.md` | Reset only **that** specialist |
| Anything else (Python logic, tests, README, releases) | **No reset** — those changes are picked up by the process restart that follows `apply_update` |

A diff that touches both a global file *and* a single agent's brief is treated as global (the global wins) — the prompt change implies every session is stale anyway. A diff that names an agent that no longer exists in `state.json` (rename or removal) is silently ignored. The helper is purely additive: every non-session field of every agent — `thread_id`, `name`, `work_dir`, `model`, `description`, `created_at`, etc. — is left verbatim.

The reset itself follows the same convention `AgentManager._load_state` already uses for placeholder UUID sanitisation: a fresh `uuid.uuid4()`, `session_started=False`, `message_count=0`.

### Why it matters

The v0.14.0 release added `_load_agent_instructions()` so interactive turns would finally see the per-agent brief from `agents/<name>.md`. The Claude Code CLI, however, bakes the system prompt at session creation time and ignores `--append-system-prompt` on `--resume`. Every agent whose Claude session pre-existed v0.14 kept running under the pre-v0.14 system prompt forever — the new instructions were silently dropped on every turn. v0.15.0 fixed this once with a one-shot migration (`0.15.0-reset-sessions-for-reminder-skill`), but the next time someone tweaked a prompt without remembering to write a new migration the same regression would have come back.

v0.15.1 makes that class of bug structurally impossible. The updater is the choke point through which every release reaches the running fleet, so anchoring the contract there means no future kael-ops release author needs to remember anything.

### `notify_fn` reports the reset summary

When the invalidation step actually resets one or more agents, `apply_update` now emits a progress notification of the form:

> Reset AI sessions for 2 agent(s): assistant, code-reviewer

This goes through the same `notify_fn` callback that already reports "Pulling latest changes..." and "Installing dependencies...", so on Telegram it lands inline in the boot summary. The side effect is no longer silent: the user always learns which agents were touched and why.

### `bot/session_lifecycle.py` — new module

Three public functions plus one constant:

- `GLOBAL_INVALIDATION_FILES: frozenset[str]` — `{"bot/config.py", "bot/ai_invoke.py"}`. Any change to one of these resets every agent.
- `agents_to_invalidate(changed_paths, known_agent_names) -> set[str] | None` — pure function, no I/O. Returns `None` for "all known agents" if a global file changed, otherwise a (possibly empty) set of named agents whose individual brief was modified. Unknown brief paths (e.g. `agents/ghost.md` for an agent that no longer exists in state) are silently dropped, which protects renames and removals.
- `reset_agent_sessions_in_state(state, agent_names) -> int` — pure mutation. Takes a `state.json`-shaped dict and a set of names (or `None` for "all"). Regenerates `session_id`, clears `session_started` and `message_count`, leaves every other field verbatim. Returns the count.
- `invalidate_sessions_for_paths(state_file, changed_paths) -> list[str]` — high-level entry point used by the updater. Reads, decides, persists, returns the sorted list of names that were reset. Never raises on missing or malformed input — the updater must always reach its restart step, and a broken `state.json` on a fresh install is a no-op success, not a hard error.

### `bot/migrations.py:_reset_sessions_for_reminder_skill` refactored

The v0.15.0 one-shot migration is kept in the registry — installs that upgrade directly from `<= 0.15.0` still need it on first boot. Its body is now a thin call to `reset_agent_sessions_in_state(state, None)` so the migration and the updater path go through the exact same reset implementation. The existing `TestResetSessionsForReminderSkill` test class is the contract, and it continues to pass unchanged after the refactor.

### `bot/updater.py:apply_update` changes

Localised to two new blocks:

1. **Pre-pull SHA capture**, after the stash and before the pull, wrapped in a try/except that logs and skips on failure. The pull continues regardless.
2. **Step 6.5: session invalidation**, after the deps install and stash pop, before the success accounting. Runs `git diff --name-only <pre_pull_sha> HEAD`, parses the path list, calls `invalidate_sessions_for_paths(STATE_FILE, changed_paths)`, and emits a `notify_fn` summary if anything was reset. Failures here are logged but never block the update — the restart still happens.

The rest of `apply_update` is byte-for-byte identical: stash → pull → migration steps → pip install → marker refresh → stash pop → **session invalidation (new)** → state accounting → return.

#### `tests/test_updater.py` — `TestApplyUpdateInvalidatesSessions` (new class, 6 tests)

End-to-end against the real `apply_update` flow with mocked `_git` and `asyncio.create_subprocess_exec`:

- Global trigger via `bot/config.py` resets every agent in the fixture; non-session fields survive.
- Per-agent brief (`agents/assistant.md`) resets only `assistant`, leaves `kael` and `code-reviewer` untouched.
- Irrelevant paths (`bot/handlers.py`, `tests/test_handlers.py`) do not rewrite `state.json`.
- Missing `state.json` (fresh install) does not break the update.
- Specialist brief (`specialists/code-reviewer.md`) resets only the specialist, with `agent_type` preserved.
- `notify_fn` callback is invoked with a `Reset AI sessions for N agent(s): …` message containing the affected agent names.

### Compatibility

- **No breaking changes.** Existing `tasks.md`, `specialists.md`, agent definitions, state files, memory entries, queue files, and `reminders.json` entries continue to work unchanged.
- **No new dependencies.**
- **The v0.15.0 one-shot migration still runs** on installs upgrading from `<= 0.15.0` (it is kept in the registry for that case). From v0.15.1 onward, the updater handles every subsequent prompt/brief change automatically.
- **Update flow now adds one `git rev-parse HEAD` (before the pull) and one `git diff --name-only <pre>..HEAD` (after the pull)** to `apply_update`. Both are local-only and complete in milliseconds.
- **No `requires_migration: true`** in the frontmatter — there are no shell migration steps to run for this release.
