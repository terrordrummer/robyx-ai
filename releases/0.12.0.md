## What's New

### Outgoing images — agents can now send pictures on request

Agents can deliver an image file to the user's chat by emitting a new response pattern:

```
[SEND_IMAGE path="/absolute/path/to/file.png" caption="optional description"]
```

- `path` must be an absolute filesystem path.
- `caption` is optional.
- Multiple `[SEND_IMAGE ...]` lines in a single response are all sent, in order.
- The platform adapter handles size limits and re-encodes the image to JPEG if it exceeds the upload cap (Telegram 10 MB, Discord 8 MB, Slack 1 GB). Agents never need to worry about byte sizes.

Typical use case: ask the Zeus Focus Stacking workspace "mandami il risultato dell'ultima iterazione" and the agent reads `research-journal.md`, identifies the latest `result.png` path, emits `[SEND_IMAGE ...]`, and the image arrives inline in the Telegram topic.

### Strict rule: explicit request only

The system prompt instructs agents to emit `[SEND_IMAGE]` **only** when the user explicitly asks for an image ("mandami il grafico", "show me the result image", "fammi vedere l'ultimo render"). Agents must not attach images proactively or as a bonus. This keeps the default conversational experience quiet and predictable.

### Automatic JPEG compression when too big

`bot/media.py` (new) exposes `prepare_image_for_upload(path, max_bytes)` which:

1. Returns the original path unchanged if it already fits.
2. Otherwise re-encodes the image as JPEG, sweeping quality from 90 → 40.
3. If quality alone is not enough, progressively downscales (100% → 25%) and repeats the quality sweep at each scale.
4. Raises `MediaError` only if the file cannot fit even at minimum quality and 25% scale, or if the source is missing or not a valid image.

RGBA images are auto-converted to RGB for JPEG compatibility. Temporary files are written under the system temp dir and cleaned up by the caller.

### Platform abstraction

New on `Platform`:

- Abstract method `async send_photo(chat_id, path, caption, thread_id) -> Any` — upload and send an image. Returns `None` on failure (failures are logged, never raised) so the handler can append an inline error notice to the text reply.
- New property `max_photo_bytes` with a per-adapter override. Defaults:
  - Telegram: 10 MiB
  - Discord: 8 MiB (conservative free-tier baseline — Nitro-boosted servers still work, they just trigger re-encoding for files between 8 MiB and their higher cap)
  - Slack: 1 GiB (generous default, Slack's file limits are very permissive)

### Handler integration

`_handle_media_commands` runs inside `_process_and_send` after the other pattern handlers (delegations, focus, workspace creation) and before the final reply send. It:

- Extracts all `[SEND_IMAGE ...]` matches.
- Strips them from the text the user sees.
- Calls `platform.send_photo` for each, in order.
- On failure (adapter returned `None` or raised), appends a one-line inline error notice to the reply text so the user is never left wondering whether the image actually arrived.

## Dependencies

- Added `Pillow>=10.0` to `bot/requirements.txt`.

## Tests

Suite grew from 581 to **601 tests** (+20). New coverage:

- `tests/test_media.py` — 7 tests for `prepare_image_for_upload`: fits-unchanged, compression loop, downscale fallback, RGBA → RGB conversion, missing file, directory, invalid image.
- `tests/test_handlers.py::TestHandleSendImage` — 6 tests: single image sent and stripped, image without caption, multiple images in order, platform returns `None` → inline error, platform raises → caught and reported, no pattern → no `send_photo` call.
- `tests/test_discord_platform.py::TestSendPhoto` + `TestMaxPhotoBytes` — 4 tests.
- `tests/test_slack_platform.py::TestSendPhoto` — 3 tests.

## Compatibility

No schema, state-file, or config changes. Drop-in upgrade from v0.11.2 — pull and restart. Agents without any updated instructions will simply ignore the new pattern (the system prompt additions are append-only).
