# Robyx 0.24.3

Robustness patch for the continuous-task dispatch loop. Makes the
history reader tolerant of step-agent schema drift, isolates per-task
crashes so one broken task cannot stall the whole fleet, and ships a
one-shot migration that normalises any pre-existing drift on disk.

## The incident

A user reported that after upgrading, a pre-existing continuous task
(`align-research`) stopped firing. Investigation showed the scheduler
was crashing on every tick with:

    KeyError: 'description'

inside `bot/continuous.py::build_step_context`. The step agent that had
executed the previous step wrote the history entry as
`{"step": N, "summary": "...", "artifact": "commit ..."}` instead of
the documented `{"step": N, "description": "...", "artifact": "..."}`.

The crash had two amplifying failures:

1. The reader used a **hard dict access** (`entry["description"]`), so
   any schema deviation was fatal.
2. The crash propagated out of the whole `_handle_continuous_entries`
   loop, so *every other* continuous task in the queue was
   undispatched on that tick — and on every subsequent tick, forever.

The user had to hand-patch the state file on disk to unblock the task.

## What changed

### `bot/continuous.py::build_step_context` is now tolerant

Each history entry is rendered defensively. The reader tries, in order,
`description` → `summary` → `artifact` → a `"(no description)"`
placeholder; non-dict entries are skipped with a warning; a missing
`step` number renders as `Step ?` rather than crashing. Drift is still
visible in logs — it is simply no longer fatal.

### Per-entry crash isolation in the scheduler loop

`_handle_continuous_entries` now wraps each iteration in a broad
`try/except`. Any unexpected error while handling one task is logged,
appended to the `errors` list, and the loop continues to the next
entry. One corrupt state file cannot block the dispatch of healthy
tasks anymore.

### Data migration: `summary` → `description`

`bot/migrations/v0_24_3.py` renames `summary` → `description` in every
history entry of every `data/continuous/<name>/state.json` on disk.
The migration is idempotent (no-op on a second run), tolerant of
unreadable state files (skip + log), and preserves entries that
already carry `description`.

### Tests

- `tests/test_continuous.py` — four new `build_step_context` cases
  covering the `summary` fallback, missing `description`/`summary`,
  missing `step` number, and non-dict entries.
- `tests/test_migration_v0_24_3.py` — new suite covering the rename,
  idempotency, existing-description preservation, unreadable files,
  non-dict history entries, empty/missing history, and the
  missing-continuous-dir edge case.

## Upgrade notes

- Operators on 0.24.2 upgrade to 0.24.3 via the built-in auto-updater or
  `git pull` + `systemctl restart robyx`. The migration framework runs
  `v0_24_3.upgrade` once on first startup after the bump — no operator
  action required.
- Tasks previously stuck on this class of drift (including hand-patched
  ones) are not at risk: the migration is idempotent and the new reader
  tolerates both the old and the new key.
