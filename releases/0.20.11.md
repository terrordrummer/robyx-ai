## Unsafe-bypass permissions across all three backends

Robyx agents run headless — nobody is sitting at the terminal to answer
"Allow this command to modify files?" prompts, so every backend must be
configured to *just do the thing*. Until now only Claude Code had an
explicit autonomous default; Codex and OpenCode were left at their own
safer defaults, which blocked agents from actually touching the disk.

This release makes every backend unsafe-bypass by default, consistent with
Claude. If a deployment explicitly wants stricter policies, the env vars
below provide a straightforward escape hatch.

### Backend-by-backend changes

**Claude Code** — unchanged behaviour, but now documented alongside the
others. Default: `--permission-mode bypassPermissions`. Override with
`CLAUDE_PERMISSION_MODE`.

> ⚠️ On Linux systems with enterprise MDM settings that force
> `permissions.disableBypassPermissionsMode: disable`, Claude Code will
> enforce the restriction regardless of what Robyx asks for. This is an
> OS/MDM-level policy outside the bot's control.

**Codex CLI** — added `--approval-policy never --sandbox danger-full-access`
to both interactive and spawned commands. Override with:
- `CODEX_APPROVAL_POLICY` (`never` | `untrusted` | `on-request` | `on-failure`)
- `CODEX_SANDBOX` (`danger-full-access` | `workspace-write` | `read-only`)

Spawned scheduled tasks always force `never` + `danger-full-access`
regardless of env config, since a scheduled task cannot be approved
interactively.

**OpenCode** — OpenCode has no CLI flag to relax permissions, only a JSON
config file. Robyx now writes a managed config at
`data/opencode-managed.json` containing:

```json
{
  "$schema": "https://opencode.ai/config.json",
  "permission": "allow"
}
```

…and exports `OPENCODE_CONFIG=<path>` so the CLI picks it up, unless the
user has already set `OPENCODE_CONFIG` explicitly (in which case we defer
to their choice). Override the permission level with `OPENCODE_PERMISSION`
(`allow` | `ask` | `deny`).
