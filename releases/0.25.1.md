# Robyx 0.25.1

Bugfix release for the auto-updater. Closes a field incident where a
prior silent `git stash pop` conflict had wedged auto-update on a
running deployment — every subsequent update attempt failed on
`git pull` ("Pulling is not possible because you have unmerged
files"), rolled back, and left the operator with no signal that the
problem was pre-existing. No persisted state schema changes; the
`v0_25_1` migration is a no-op.

## The incident

A user on `v0.24.3` triggered auto-update to `v0.25.0` and saw:

    Update failed: git pull --ff-only failed: error: Pulling is not
    possible because you have unmerged files.
    Rolled back to v0.24.3.

Inspection of the deployment's working tree found:

- HEAD on `main` at `v0.24.3`, working tree clean.
- **Index** in an unmerged state on `bot/continuous.py` (stages 1/2/3
  populated) with four other files staged as "resolved".
- **No** `MERGE_HEAD`, `REBASE_HEAD`, or `CHERRY_PICK_HEAD` — nothing
  actually in progress.
- A stash from months earlier (back when the deployment was on
  `v0.24.2`) containing the user's local modifications.

The diagnosis: a previous auto-update (some time between 0.24.2 and
0.24.3) had its post-update `git stash pop` conflict. The stash was
kept, but the conflict was logged at a low-signal `WARNING`, the
update wrapper treated the overall run as successful, and the index
was left in a broken-merge state. Every subsequent update attempt ran
into the same index state on its pull step, rolled back, and emitted
the same unhelpful error.

## What changed

### Pre-flight gate in `apply_update`

`bot/updater.py::_preflight_git_state` (new) runs before the first
stash. It refuses to start the update if:

- `git ls-files --unmerged` reports any path — the index has
  leftover conflict stages from a prior interrupted operation;
- `.git/MERGE_HEAD`, `.git/CHERRY_PICK_HEAD`, `.git/REVERT_HEAD`,
  `.git/rebase-apply/`, or `.git/rebase-merge/` exists — a real
  in-progress merge / cherry-pick / revert / rebase is underway.

The error message tells the operator exactly which files are unmerged
(or which operation is mid-flight) and how to recover
(`git add` + commit, or `git reset --mixed HEAD` for stale changes, or
`git <op> --abort`).

The gate runs BEFORE the stash and BEFORE the snapshot, so a pre-flight
failure leaves the working tree, the stash, and the snapshot list
exactly as they were. No side effects from a refused update.

### Louder `_safe_stash_pop` on conflict

The post-update stash pop (step 7 of the safety structure) still runs
best-effort, but when it leaves unmerged paths behind the log level
escalates from `WARNING` to `ERROR`, the message names each affected
file, and it explicitly tells the operator that the next auto-update
will be blocked by the pre-flight gate until the conflict is resolved.
The previous single-line `WARNING` without file list is what let the
original incident hide.

## Tests

Three new regression tests in `tests/test_updater.py`:

- `test_preflight_refuses_when_unmerged_paths_exist` — reproduces the
  exact field condition (unmerged index, no in-progress merge) and
  asserts pre-flight refuses before stashing or pulling.
- `test_preflight_refuses_mid_rebase` — asserts the gate catches a
  real in-progress rebase (via `.git/rebase-merge/`).
- `test_safe_stash_pop_surfaces_unmerged_paths` — asserts conflicted
  stash pop logs at `ERROR` with the file list.

Full suite: **1700 passed, 1 skipped**.

## Upgrade notes

- Operators on 0.25.0 upgrade to 0.25.1 via the built-in auto-updater
  or `git pull` + `systemctl restart robyx`. The migration framework
  runs `v0_25_1.upgrade` as a no-op.
- If your deployment is currently wedged with unmerged index entries
  from a prior failed update, **resolve the index before the next
  auto-update**:
  1. `cd <robyx install>`
  2. `git status` — see which files are unmerged.
  3. If the leftover staged changes are stale (content the update
     already shipped), `git reset --mixed HEAD` clears them.
  4. If they are real local modifications, resolve the conflict
     normally (`git add <file>` each), then commit or stash them.
  Once `git status` reports no unmerged paths, the next auto-update
  will proceed past the new pre-flight gate.
