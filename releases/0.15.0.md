## Reminders as a universal skill

A focused release that promotes reminders from a per-agent improvisation to a **first-class skill available to every interactive agent in KaelOps** — Kael in Headquarters, every workspace agent, every specialist, and any agent currently in focus mode. The skill is exposed through a single declarative pattern, parsed centrally by the bot, and backed by the existing Python reminder engine that has been wired in since v0.12.4.

This release also delivers the most important infrastructure piece around it: a migration that **forces every agent's AI-CLI session to be regenerated on first boot**, so the new system prompt with the new skill actually reaches the existing fleet — not just future installs.

---

### The new pattern: `[REMIND ...]`

Any interactive agent can schedule a plain text message to be delivered to its own topic at a precise future time by emitting the new pattern in its reply:

```text
[REMIND in="2m" text="⏰ sei un figo"]
[REMIND in="1h30m" text="meeting prep"]
[REMIND at="2026-04-09T09:00:00+02:00" text="📅 dentist appointment"]
[REMIND in="2d" text="📦 ordina cartucce stampante" thread="903"]
```

Attributes:

| attribute | required | description |
|---|---|---|
| `text` | yes | The message that will be delivered verbatim. Unicode and emoji allowed. |
| `at` | one of | ISO-8601 datetime with explicit timezone offset. Past values are tolerated up to 60 s (clock skew). |
| `in` | one of | Compact duration: `90s`, `2m`, `1h30m`, `2d`. Up to 90 days. |
| `thread` | no | Integer thread id. Defaults to the topic the agent is currently living in — agents never need to know their own thread id. |

The bot fills `thread` in automatically from the live message routing context, builds the reminder dict, and appends it to `data/reminders.json` via the new thread-safe `append_reminder()` helper. The pattern is stripped from the user-visible reply. Validation failures (missing `text`, both `at` and `in`, malformed datetime, past `at`, invalid duration, over-90-day duration) surface as **inline notices** in the reply — never as silent drops. Multiple `[REMIND ...]` patterns per response are allowed, e.g. one nudge the day before plus one on the day.

The Python reminder engine (`bot/reminders.py`) is unchanged: it polls every 60 seconds via `bot/bot.py:reminders_job`, fires due entries directly through the platform adapter without spinning up an LLM, and is idempotent (`status: pending → sent`) and late-firing on bot restart (entries whose `fire_at` is in the past are dispatched on the next polling tick after recovery).

### Why a pattern, not file-writing

The previous `## Scheduling Tasks` section in `WORKSPACE_AGENT_SYSTEM_PROMPT` asked workspace agents to hand-write JSON to `data/timed_queue.json` — including UUIDs, atomic temp-file rename, ISO timestamps, and their own `thread_id` and `agent_file`. That was brittle for haiku-class models, impossible for the orchestrator (which never executes project work from Headquarters), and wasted a full agent re-invocation just to deliver a text message. A single declarative pattern parsed centrally by the bot eliminates every one of those failure modes and is dramatically cheaper per fire (zero LLM cost vs. a full sub-agent run).

The Timed Task Queue is **not** removed — it remains the right tool when an agent genuinely needs to be **re-invoked at a future time to do work**, not just deliver a message. The workspace prompt now leads with `[REMIND]` for the common case and reserves `timed_queue.json` for the rare "re-run me at T to perform actual work" scenario.

### The migration that delivers the new skill to existing agents

This is the part that matters for upgrades.

In v0.14.0, `ai_invoke._load_agent_instructions()` started appending `agents/<name>.md` to the system prompt on every interactive turn. The Claude Code CLI, however, **bakes the system prompt into the session at session creation time** and ignores `--append-system-prompt` on `--resume`. So any agent whose Claude session pre-existed v0.14 has continued running under whatever system prompt it was born with — the new instructions are silently dropped on every turn.

A symptom we observed against the v0.14 fleet: asking the personal assistant on Telegram thread 903 to "remind me in 2 minutes" returned "reminders are session-based, the cadence is hourly". The assistant was running under the pre-v0.14 system prompt, never saw the v0.14 brief from `agents/assistant.md`, and improvised. The same blind spot would have hidden v0.15's new `## Reminders` section from every existing agent.

The new migration **`0.15.0-reset-sessions-for-reminder-skill`**:

1. Reads `data/state.json` directly (no `AgentManager` instantiation, to avoid touching `_setup_orchestrator` and the placeholder-UUID sanitiser).
2. For every agent in `state["agents"]`, regenerates `session_id = uuid.uuid4()`, sets `session_started = False`, sets `message_count = 0`. Leaves `thread_id`, `name`, `work_dir`, `model`, `description`, `created_at`, and every other field untouched.
3. Persists state.

The next interactive turn for each agent creates a brand-new Claude session with the v0.15 system prompt — and finally bakes in the universal `## Reminders` section. The reset matches the convention `AgentManager._load_state` already uses when sanitising placeholder UUIDs (`session_id = str(uuid.uuid4())`, `session_started = False`, `message_count = 0`).

Idempotency is delegated to the migrations framework: `run_pending` records this migration in `data/migrations.json` and never re-runs it. Fresh installs with no `state.json` are a no-op success. Boot notification on Telegram lists the migration inline the first time the new build comes up (`bot.py:boot_notify`), so the user gets a clear signal that the upgrade has propagated to the live fleet.

### System prompts: one section, three places

A new identical `## Reminders` section is now present in **all three** interactive system prompts in `bot/config.py`:

- `KAEL_SYSTEM_PROMPT` — placed after `## Sending Images`, before `## Rules`. Kael can now schedule reminders for the user directly from Headquarters without delegating.
- `WORKSPACE_AGENT_SYSTEM_PROMPT` — placed after `## Sending Images`. The existing `## Scheduling Tasks` section is preserved but reframed at the top to point new readers at `[REMIND]` for the common case.
- `FOCUSED_AGENT_SYSTEM_PROMPT` — placed after `## Sending Images`. Reminder scheduling now works in focus mode too, where previously the agent had no way to schedule anything.

A new test class (`TestSystemPromptsHaveRemindersSection` in `tests/test_reminders.py`) asserts that the literal string `## Reminders` and `[REMIND` appear in all three prompts. This is the regression guard against future drift where one prompt forgets the universal skill.

The "anticipo intelligente" table and the `memory.md` workflow are kept verbatim — they are personality, not infrastructure.

### `bot/reminders.py` — small but important hardening

- New `append_reminder(reminders_file, entry)` helper. Acquires the existing module `_lock`, loads the file (creating `[]` if missing), appends, atomically rewrites. The caller in `bot/handlers.py` uses this and never touches the file directly.
- `_save` is now atomic: it writes to a `.tmp` file in the same directory and `replace`s the target. Both `check_reminders` and `append_reminder` go through the same lock, so the new flow inherits race safety for free.
- The engine's `check_reminders` is **unchanged** — its existing per-reminder `thread_id` routing (`bot/reminders.py:66-73`) was already generic and just needed to be exposed to all agents.

### Compatibility

- **No breaking changes.** Existing `tasks.md`, `specialists.md`, agent definitions, state files, memory entries, and `timed_queue.json` entries continue to work unchanged.
- **`reminders.json` schema is unchanged.** Entries already in the file from v0.12.4+ continue to fire normally; the engine still reads `id`, `message`, `fire_at`, `thread_id`, `status`, `created_at`. The new `append_reminder` helper writes the same shape.
- **The Timed Task Queue is unchanged.** Existing periodic and one-shot entries in `data/timed_queue.json` keep dispatching at the configured interval. The workspace prompt is reframed but continues to document the JSON-write path for the work-at-T case.
- **Migration**: the session reset runs once at the next boot. Every agent will create a fresh AI-CLI session on its next interactive turn (one extra cold start per agent — ~2-5 s of additional latency on the first message after upgrade). Subsequent turns resume normally.
- **No new dependencies.** No additions to `bot/requirements.txt`.
