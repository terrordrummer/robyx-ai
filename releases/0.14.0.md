## Headquarters, backend-aware models, OpenCode sessions, and Telegram resilience

A consolidation release that bundles every fix from PR #2 (Daniel Metzing /
`sleep-update`) on top of the v0.13.0 timed task queue plus the v0.12.4
hotfix that landed on `main` in parallel (Python reminder engine and
`AI_TIMEOUT` raised to 7200s for long R&D runs), and rebrands the control
room from "Command Bridge" to **Headquarters**. Nothing in your existing
`tasks.md`, agents, specialists, state files, or memory needs to change —
every behaviour change is either backwards-compatible or carried through
by the migration framework on first boot.

---

### Headquarters — the control room is renamed

Kael's home is now called **Headquarters** instead of "Command Bridge". The
rename runs once on the first boot after upgrading via a new migration
(`0.14.0-rename-command-bridge-to-headquarters`):

- **Telegram**: the General topic of the forum supergroup is renamed to
  `Headquarters` via `editGeneralForumTopic` (verbatim, mixed case allowed).
- **Discord** and **Slack**: the control channel is renamed to the slug
  `headquarters` because their channel names must be lowercase and
  dash-separated.
- The historical `0.12.1-rename-main-to-command-bridge` migration is kept in
  the registry so fresh installs that have never run it still pass through
  it before the new one — both run in registration order, so a brand-new
  clone ends up correctly named in a single boot.

All user-visible strings in `bot/i18n.py`, the orchestrator system prompt
(`KAEL_SYSTEM_PROMPT`), `ORCHESTRATOR.md`, `SCHEDULER.md`, and `README.md`
now say "Headquarters". Historical release notes for v0.12.x are left
untouched as a record of the previous naming.

### Telegram polling resilient to macOS sleep/wake

The previous default `run_polling` would silently hang for minutes after a
macOS sleep/wake cycle: the underlying TCP connection was already dead but
neither side detected it until the next write actually failed. The bot
*looked* alive but never answered until the OS gave up.

`bot/bot.py` now passes an explicit polling configuration via the new
`telegram_polling_kwargs()`:

| key | value | reason |
|---|---|---|
| `timeout` | `10` | long-poll cycle hard-capped at 10 s |
| `read_timeout` / `write_timeout` / `connect_timeout` / `pool_timeout` | `15` | every HTTP step times out within 15 s |
| `poll_interval` | `1.0` | sub-second recovery loop |
| `bootstrap_retries` | `-1` | retry the initial connection forever |
| `drop_pending_updates` | `True` | skip the backlog accumulated while sleeping |

A Python 3.13 guard also makes sure an event loop exists before
`run_polling` starts, since recent CPython versions stopped creating one
eagerly on the main thread.

The bot now recovers within ~15 seconds of any sleep/wake event, and the
new test class `TestTelegramPollingKwargs` will scream if any of those
values regress.

### Telegram control room is the General topic again

The Telegram General topic of a forum supergroup must be addressed with
`message_thread_id=0`. The previous hard-coded `1` was rejected by recent
Bot API versions, so every scheduler / boot / update notification was
*silently* dropped — the bot looked online but Headquarters stayed quiet.

`TelegramPlatform.control_room_id` now returns `0`, and `bot.py` sources
the thread id for `scheduler_job`, `update_check_job`, and `boot_notify`
from `plat.control_room_id` instead of the old `TELEGRAM_MAIN_THREAD_ID =
None` constant. New tests
(`TestSchedulerJobRoutesViaControlRoom`, `TestControlRoomId`) lock the
behaviour in.

`Platform.control_room_id` was widened to `int | None` so future adapters
that have no thread concept can return `None` without violating the type.

### `send_message` bypasses python-telegram-bot

`TelegramPlatform.send_message` now POSTs directly to the Bot API via
`httpx`. PTB's `Bot.send_message` had been intermittently unreliable for
control-room sends in forum chats (silent drops, occasional 60 s hangs
after sleep/wake) — using the same payload that direct `curl` probes use
makes the failure mode predictable, easy to time out (30 s), and easy to
test. New tests cover the happy path, the `thread_id=None` branch, and
API-level error propagation.

### Detached workspaces self-heal on boot

A workspace becomes *detached* when its row in `tasks.md` lists `-` as the
Thread ID (e.g. the topic was manually closed, or the agent was created on
another machine). The new `topics.heal_detached_workspaces(manager,
platform=…)`:

1. Walks every live workspace agent.
2. For each one without a `thread_id`, creates a fresh forum topic on the
   active platform.
3. Updates the agent in-place via `manager.add_agent(...)`, persisting the
   new model preference along with the new thread id.
4. Rewrites the row in `tasks.md` so the next boot sees the healed state
   (helpers `_update_task_thread_id` / `_update_specialist_thread_id` /
   `_update_table_thread_id`).
5. Posts a welcome message in the freshly-attached topic.

Hooked into Telegram boot through `boot_notify` *before* the boot message
goes out, so the boot summary reflects the freshly-attached channels.
Failures are logged but never fatal — a workspace that cannot be re-attached
just stays detached.

### Backend-aware model preferences (`models.yaml`)

Workspaces, specialists, and scheduled tasks can now express their model
intent as a **semantic alias** (`fast` / `balanced` / `powerful`) or as a
**role** (`orchestrator` / `workspace` / `specialist` / `scheduled` /
`one-shot`). KaelOps resolves the alias at invocation time into the
concrete model id understood by the active backend.

#### New file: `models.yaml`

```yaml
defaults:
  orchestrator: balanced
  workspace: balanced
  specialist: powerful
  scheduled: fast
  one-shot: fast

aliases:
  fast:
    claude: haiku
    codex: gpt-5-mini
    opencode: openai/gpt-5-mini
  balanced:
    claude: sonnet
    codex: gpt-5
    opencode: openai/gpt-5
  powerful:
    claude: opus
    codex: gpt-5.4
    opencode: openai/gpt-5.4
```

#### New module: `bot/model_preferences.py`

`resolve_model_preference(model, backend, role)` is the single resolver. Its
contract:

1. If *model* is set, use it (after legacy alias normalisation —
   `haiku`/`sonnet`/`opus` are silently mapped to `fast`/`balanced`/`powerful`
   so existing `tasks.md` and `specialists.md` rows keep working).
2. Otherwise, fall back to the configured default for *role*.
3. Look the result up in `aliases[<alias>][<backend>]`. If found, return
   the concrete model id; otherwise return the alias unchanged — power
   users can put a literal `openai/gpt-5.4-preview` in a tasks row and have
   KaelOps pass it through verbatim.

Three layers of fallback in `bot/config.py`:

1. `models.yaml` (preferred — versioned, shared).
2. `AI_MODEL_DEFAULTS` / `AI_MODEL_ALIASES` env vars (JSON; legacy).
3. Hard-coded `DEFAULT_MODEL_*` (always-safe baseline).

A brand-new clone with no `models.yaml`, no env vars, and no PyYAML still
boots cleanly.

#### Wiring

- `bot/scheduler.py:spawn_task` and `bot/timed_scheduler.py:dispatch_task`
  both route the queued model through `resolve_model_preference` so
  scheduled and timed tasks honour aliases.
- `bot/ai_invoke.py:_invoke_ai_locked` resolves
  `model or agent.model` against the agent's role default — so interactive
  turns finally honour the per-agent preference instead of silently falling
  back to `sonnet`.
- `Agent.model` is a new dataclass field, persisted in
  `data/state.json` and propagated by
  `topics.create_workspace` / `topics.create_specialist` /
  `AgentManager.add_agent(model=...)`. Re-adding the same agent without a
  model arg preserves the previous preference (so `heal_detached_workspaces`
  cannot accidentally erase it).

#### New dependency

`PyYAML>=6.0` in `bot/requirements.txt`. The bootstrap installer will pick
it up automatically on the next boot, just like Pillow did in v0.12.0.

### OpenCode session persistence and JSON output

`OpenCodeBackend` was a single-shot wrapper around `opencode run`: every
turn started a fresh session, so multi-turn conversations had no context.
Now:

- `supports_sessions()` returns `True`.
- `can_resume_session(sid)` filters out KaelOps' generic UUID — only
  OpenCode's native `ses_…` ids are passed to the CLI via `--session`.
- Commands now include `--format json`, so the CLI emits structured output
  (NDJSON or a single JSON blob — the parser handles both).
- `parse_response` walks the payload, extracts `result` / `text` /
  `message.content`, and recursively looks for `sessionID` /
  `sessionId` / `session_id` / nested `session.*` keys.
- `parse_response` returns `{text, session_id}` for backends with metadata;
  legacy backends still return a plain string.
- `_normalize_backend_response` in `bot/ai_invoke.py` hides the
  string-vs-dict difference from the rest of the invocation pipeline.
- The captured session id is persisted on `agent.session_id` and
  `manager.save_state()` after every successful turn, so the conversation
  resumes seamlessly across messages **and** across bot restarts.
- `AIBackend.parse_response` is now typed `str | dict[str, Any]`. Existing
  Claude / Codex backends are unchanged.

Since OpenCode has no `--system-prompt` flag, KaelOps now wraps the
orchestrator system prompt inside `<system_instructions>` /
`<user_message>` tags inside the user message itself. The CLI sees a single
message; the model sees clearly delimited instructions.

### Per-agent instructions on interactive turns

Workspace and specialist agents have a markdown brief at
`agents/<name>.md` / `specialists/<name>.md`. The interactive invocation
path used to ignore them — only scheduled spawns saw the full brief, so
chatting with a workspace agent gave generic answers while letting a
scheduled run of the same agent execute correctly.

`ai_invoke._load_agent_instructions(agent)` now appends the markdown
content to `WORKSPACE_AGENT_SYSTEM_PROMPT` whenever the agent type is
`workspace` or `specialist`. Empty files and missing files are silently
treated as "no extra instructions".

### Compatibility

- **No breaking changes.** Existing `tasks.md`, `specialists.md`, agent
  definitions, state files, memory and timed-queue entries continue to
  work unchanged.
- **Legacy model names** (`haiku` / `sonnet` / `opus` in your tasks rows)
  keep working — they are silently remapped to `fast` / `balanced` /
  `powerful` by the resolver.
- **`models.yaml` is optional.** If absent, KaelOps falls back to
  `AI_MODEL_DEFAULTS` / `AI_MODEL_ALIASES` env vars, then to the
  hard-coded defaults — identical to the previous behaviour.
- **PyYAML** is the only new dependency. The bootstrap installer
  (`bot/_bootstrap.py`) reinstalls dependencies automatically on the next
  boot.
- **Migration**: the Headquarters rename runs once at the next boot. If
  your bot lacks the rename permission on the General topic the migration
  records as `failed` and is never retried — the channel just stays named
  "Command Bridge", everything else still works.
