## Updater hardening — root-cause fix for "not on a branch" trap

v0.20.21 added a defensive re-attach step that recovered from detached
HEAD at update time. This release attacks the root cause: the rollback
path no longer detaches HEAD in the first place. Combined with a
stronger pre-pull check (the branch must actually be ``main``, not just
"any branch"), the working tree is now guaranteed to stay attached to
``main`` across any sequence of successful and failed updates.

### Fixed — rollback keeps HEAD on ``main``

**Bug**: every rollback point used ``git checkout v<current>`` (7 call
sites: pip failure, migration failure, migration timeout, smoke-test
failure, exception handler, catastrophic path). That command moves the
working tree to the tag **but detaches HEAD**. Any subsequent
``git pull --ff-only`` then aborted with "You are not currently on a
branch" — the install was stuck until a human ran ``git checkout main``.

**Fix**: new helper ``_rollback_code_to(tag)`` that does
``git checkout main`` followed by ``git reset --hard v<tag>``. The tree
ends up at the same content as before, but HEAD is still the ``main``
branch, so the next update cycle can fast-forward cleanly. All 7
rollback call sites now go through the helper.

### Changed — pre-pull branch check targets ``main`` explicitly

**Before (v0.20.21)**: the updater only verified that HEAD was not
detached. If the operator had manually checked out a feature branch or
an older tag-as-branch, the pull would run against that branch — not
``main``.

**After**: we parse the current branch with
``git symbolic-ref --short HEAD`` and compare against the string
``main``. If it differs (detached or any other branch), we switch to
``main`` before pulling. The notification message distinguishes the two
cases so operators can see what happened:

- Detached: ``"Detached HEAD detected — reattaching to main"``
- Wrong branch: ``"On branch 'X', update targets 'main' — switching"``

Local changes were already stashed one step earlier, so there is
nothing to lose.

### Tests

984 passed, 1 skipped (+2 regression tests):

- ``TestApplyUpdate.test_feature_branch_switches_to_main`` — a
  non-``main`` branch is switched away from before the pull.
- ``TestApplyUpdate.test_already_on_main_no_extra_checkout`` — the
  normal path (already on main) never issues a redundant ``checkout
  main``.

  to assert ``reset --hard v<tag>`` instead of the old detached
  ``checkout v<tag>``.

### Migration

None. ``bot/migrations/v0_20_22.py`` is a no-op. No data/state changes.

### Upgrade path

Installs that already applied v0.20.21 auto-upgrade normally. This is
the first release where the rollback guarantees are enforced
structurally — no more manual recovery after a bad release.
