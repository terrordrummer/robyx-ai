## Persistent httpx client + the real fix for slow Headquarters typing

The "typing indicator doesn't appear immediately when I send a message
in Headquarters" bug had survived four prior fixes (v0.20.6, .7, .8,
.9). Each one addressed a real symptom but missed the underlying
cause: **every Telegram API call was constructing a brand-new
`httpx.AsyncClient`**, which means every `send_typing` did a cold
DNS lookup + TCP handshake + TLS handshake before the request even
left the box. On a warm cache that's ~50ms; cold (after macOS sleep,
network roaming, or simply a quiet period) it's 200-500ms — long
enough that the user perceives "no typing".

### Fixed

- **`bot/messaging/telegram.py`** — `TelegramPlatform` now owns a
  single persistent `httpx.AsyncClient`, lazily created on first use,
  with a keepalive pool sized for api.telegram.org (4 idle, 8 max,
  300s expiry). `send_typing`, `send_message`, `send_photo`,
  `send_to_channel`, `create_channel`, `close_channel` all reuse it.
  An `aclose()` method closes the client cleanly at shutdown or for
  test teardown.

- **`bot/handlers.py`** — The early `send_typing` in `handle_message`
  was wrapped in `try/except Exception: pass`, so any failure (network
  blip, rate limit, DNS hiccup) silently produced no typing indicator.
  Now logs at `WARNING` with chat / thread context. Also dispatched as
  a background `asyncio.create_task` so the handler doesn't block on
  Telegram's roundtrip — agent invocation starts in parallel with the
  typing-action HTTP call.

### Added

- **`tests/test_handlers.py`** — Three regression tests that pin down
  the contract for the user-reported bug:
  - `test_typing_fires_for_headquarters_general_topic` — a message
    in Headquarters (`thread_id=None`) triggers `send_typing(chat_id,
    None)`, the exact wire format Telegram needs to display typing in
    the General topic of a forum supergroup.
  - `test_typing_does_not_block_message_processing` — a slow
    `send_typing` does not delay the agent invocation; both run in
    parallel.
  - `test_typing_failure_is_logged_not_silenced` — a `send_typing`
    that raises is logged at WARNING rather than swallowed silently
    (silent swallowing is how the original bug survived four prior
    "fixes").

- **`tests/test_telegram_platform.py`** — Five tests for the
  persistent client behaviour:
  - `_get_client` returns the same instance across calls.
  - 5 sequential `send_typing` calls construct exactly 1
    `AsyncClient` (the regression guard against re-introducing
    per-call client creation).
  - `send_typing` omits `message_thread_id` for General
    (`thread_id=None`) and includes it for forum topics.
  - `aclose` closes the cached client and clears the cache.

### Result

971 tests pass, 0 fail. The next time someone says "typing doesn't
show in Headquarters", we have a regression-test suite that pins down
the three contracts above — no more silent failures.

### No migration needed

Pure code fix. The bootstrap migration `v0_20_16.py` is a no-op.
