# Robyx 0.23.0

Feature `005-unified-workspace-chat` — **unified workspace chat for
scheduled & continuous tasks**. Minor release.

## User-visible change

Continuous tasks no longer open a dedicated sub-topic on Telegram / Discord
/ Slack. Every task — continuous, periodic, one-shot, reminder — now
reports into the **parent workspace chat** with a type-specific icon
marker, and the user controls the full lifecycle by talking to the primary
workspace agent in natural language.

Before 0.23.0: creating a continuous task opened a `🔄 <name>` sub-topic;
scheduler reports landed there; user interruptions lived there too; the
workspace chat and the sub-topic were two places to watch.

After 0.23.0: one chat. One agent. One conversation.

## Highlights

### Unified delivery with icon markers

Every scheduled delivery now carries a type-specific prefix applied at a
single chokepoint in the delivery layer (agents do not format this
themselves):

- 🔄 continuous
- ⏰ periodic
- ⏰ _(periodic aliases)_
- 📌 one-shot
- 🔔 reminder

Primary-agent conversational replies are **never** marked — markers are
reserved for scheduler-driven output.

### Primary-agent lifecycle control (natural language)

Talk to the primary workspace agent to list, inspect, stop, pause,
resume, or read the plan of any task in the workspace:

- "lista task" → grouped summary with icons
- "stato daily-report" → detailed status
- "ferma daily-report" → terminal (marks completed + cancels queue entry)
- "pausa daily-report" → continuous only
- "ripristina daily-report" → resume from paused / rate-limited
- "dimmi il piano di daily-report" → reads `plan.md`

Ambiguous queries (multiple tasks match a substring) always produce a
disambiguation prompt. The primary agent **never** silently guesses.

### Macro leak closure

The `[CREATE_CONTINUOUS …]` / `[CONTINUOUS_PROGRAM] …[/CONTINUOUS_PROGRAM]`
strip now happens at two redundant chokepoints:

- `bot/continuous_macro.apply_continuous_macros` (existing, dispatches)
- `bot/handlers._send_response` via the new `strip_control_tokens_for_user`
  canonical helper (defense-in-depth; closes spec 004 P1 on every
  interactive code path including TTS and platform-specific renderers)

The scheduled path in `bot/scheduled_delivery._clean_result_text` also
uses the same canonical helper.

### Secondary-agent knowledge parity

When the scheduler spawns a continuous step, the secondary agent now
receives:

- the parent workspace's `agents/<name>.md` instructions (same as the
  primary)
- the task-specific `data/continuous/<name>/plan.md`
- the current `state.json` fields (program, history, step state)

This prevents behavioural drift between primary and secondary.

### Idempotent migration (`bot/migrations/v0_23_0.py`)

On the first 0.23.0 boot:

- every pre-existing `data/continuous/<name>/state.json` has its
  `workspace_thread_id` repointed at the parent workspace chat;
- the old sub-topic id is preserved in `legacy_workspace_thread_id`;
- a `plan.md` is generated for tasks that never had one;
- the legacy sub-topic is best-effort closed via
  `Platform.close_channel()` (with a fallback "migrato" notice where
  close is unsupported);
- a single transition notice is posted in the parent workspace chat:
  `"🔄 [<name>] migrato — da ora riporto qui."`;
- idempotency is enforced by both a per-task `migrated_v0_23_0`
  timestamp and a process-wide `data/migrations/v0_23_0.done` marker.

Running the migration a second time produces **zero** additional
notices, state changes, or `close_channel` calls.

## Data / schema changes

`data/continuous/<name>/state.json` gains three additive fields at
migration time:

- `legacy_workspace_thread_id` — audit record of the old sub-topic id
- `plan_path` — relative path to the per-task plan artifact
- `migrated_v0_23_0` — ISO-8601 UTC timestamp of the migration

`workspace_thread_id` is rewritten to the parent workspace thread.

The scheduler queue schema is unchanged.

## Rollback

Forward-only migration. To roll back manually:

1. Stop the bot.
2. Restore each `data/continuous/<name>/state.json` from a pre-upgrade
   backup, OR copy `legacy_workspace_thread_id` back into
   `workspace_thread_id` and remove the `legacy_workspace_thread_id`,
   `plan_path`, and `migrated_v0_23_0` fields by hand.
3. Delete `data/migrations/v0_23_0.done`.
4. Checkout `v0.22.2` and restart.
