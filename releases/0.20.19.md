## P3 performance + P4 documentation — closing the review plan

This release closes the remaining two buckets of the architecture
review started in v0.20.17. No behavioural regressions, no schema
changes, one new test file, and two new doc pages.

### Added (P3 — performance)

- **`bot/ai_invoke.py`** — `_load_agent_instructions()` now caches
  the assembled payload by file mtime. The previous code re-read
  the agent brief from disk on every turn; agents with large
  briefs paid that cost at every invocation. Cache is keyed by
  absolute path, invalidated automatically when mtime changes (on
  edits from chat, `/reset`, migrations, or anything that touches
  the file on disk). Memory overhead is negligible — one string
  per workspace agent.

- **`bot/ai_invoke.py`** — system-prompt assembly now hard-truncates
  at 50 000 words (~65k tokens). Above that the context window is
  effectively consumed before the user's message even arrives,
  which produced opaque "model refused to answer" failures.
  Truncation clips from the end (memory context is appended last
  and therefore first to go) and stamps a visible marker in the
  prompt so the agent can see that content was dropped. The
  existing 30 000-word WARN threshold stays in place as an
  early-warning signal.

- **`bot/scheduler.py`** — queue-size watchdog: when `queue.json`
  crosses 500 entries, the next mutation emits a one-shot WARN so
  operators can prune or archive before scheduler-tick cost starts
  creeping up. Resets below 250 entries so a second warning fires
  if the queue grows again later.

### Skipped (P3)

- **In-memory queue index** — evaluated and rejected for this
  cycle. At the current scale (< 100 entries, one full-list scan
  per 60-second tick) the O(n) cost is not measurable; adding an
  index requires maintaining four invariants under
  `_queue_mutex()` for no practical gain. The size watchdog above
  gives us a triggered escalation path if a deployment grows
  beyond that envelope.

### Added (P4 — DX / docs)

- **`docs/data-directory.md`** (new) — full contract for every
  file under `data/`: who writes it, what happens if you delete
  it, whether it carries agent registry / queue / memory /
  continuous-task state, and how backup and recovery work via
  the updater's snapshot mechanism. Also a "what is NOT stored
  here" section so new users stop wondering where code and
  configuration live. Linked from the README documentation table.

- **`docs/scheduler.md`** — new **Timing precision** section
  documenting the single-tick cadence, the offline-recovery
  guarantee (no event lost, only delayed), the drift-free
  `_next_run_after()` semantics for periodic tasks, and the
  fact that continuous tasks are state-file-driven rather than
  claim-based. This is information users repeatedly asked for
  in practice — now it is one page lookup instead of a code
  dive.

- **`tests/test_integration_p0_p3.py`** (new) — 8 regression
  tests pinning down the specific contracts that the P0+P1+P3
  passes touched:
  - fcntl sidecar lockfile is created on first queue mutation,
  - concurrent `save_queue` round-trips stay consistent under
    the mutex,
  - stale claims on removed entries log at INFO (benign) while
    dispatched-but-unrecorded claims log at ERROR (risky),
  - reminders older than `REMINDER_MAX_AGE_SECONDS` are marked
    `failed` with `failure_reason="expired"`,
  - `cleanup_stale_locks_on_startup` removes lockfiles pointing
    at dead PIDs,
  - instructions cache serves the cached payload when mtime has
    not changed,
  - instructions cache reloads when mtime advances.

### Tests

979 passed, 1 skipped, 3 warnings. +8 from the new integration
file. No existing test was modified.

### No migration needed

All P3 changes are pure code-path optimisations; no on-disk
schema changes. `bot/migrations/v0_20_19.py` is a no-op required
only for chain continuity.

### Plan closed

Every bucket of the architecture review plan
(`~/.claude/plans/twinkly-hugging-map.md`) is now implemented or
explicitly rationalised. Two items were skipped with documented
reasoning (single-pass pattern extraction in v0.20.18, in-memory
queue index here); every other recommendation landed either as
behaviour fix, docs, or tests. Net result across the four
releases:

- `config.py`: 604 → 233 lines (−61%)
- Scheduler: two concurrency races closed, fcntl multi-instance
  protection added, continuous-task deadlock window eliminated,
  shared spawn helper extracted
- Robustness: heartbeat, retry, orphan cleanup, stale-lock scan,
  reminder max-age
- Performance: instructions cache, prompt truncation, queue-size
  watchdog
- Docs: scheduler timing contract, data-directory contract
- Tests: 972 → 980 (+8 integration, no regressions)
