# Robyx 0.24.0

**Continuous-task lifecycle hardening.** Three coordinated fixes so the
primary workspace agent never again replaces, duplicates, or needlessly
interrupts a continuous task.

## What this release is about

In 0.23.0 the primary workspace agent learned to list, stop, pause,
resume, and read the plan of any continuous task in the workspace
(`[LIST_TASKS]`, `[STOP_TASK]`, `[PAUSE_TASK]`, `[RESUME_TASK]`,
`[GET_PLAN]`). But three gaps remained:

1. **The step agent did not honour `checkpoint_policy`.** You could
   configure a task with `on-milestone` or `on-uncertainty`, and the
   step agent would still stop at arbitrary points because the policy
   was never injected into its prompt.
2. **The primary agent was blind to ongoing tasks.** When you replied
   in the workspace chat to a task that was `awaiting-input`, the
   primary agent had no record that a task even existed — it read your
   reply as a fresh request, and often spawned a *new* continuous task
   with a slightly different scope, leaving the original orphaned.
3. **There was no path to modify a task in place.** If you wanted to
   adjust the scope of a running task, the primary agent's only
   "clean" tool was `[CREATE_CONTINUOUS]` — which creates a duplicate,
   not an update.

0.24.0 closes all three.

## Highlights

### 1. `checkpoint_policy` is now binding

The step agent's prompt (`templates/CONTINUOUS_STEP.md`) now includes an
explicit `## Checkpoint Policy` section with per-policy rules:

- `on-demand` (default) — never stop on your own initiative. Fail into
  `status: error` if a step is impossible; never into `awaiting-input`
  unless the user configured a stop point.
- `on-uncertainty` — stop only for genuinely blocking ambiguity. "A or
  B both look fine" is *not* uncertainty.
- `on-milestone` — stop only at milestones declared in the plan.
- `every-N-steps` — stop only at step numbers that are multiples of N.

The scheduler substitutes `{{CHECKPOINT_POLICY}}` into the prompt from
`state.program.checkpoint_policy` on every dispatch.

### 2. The primary agent now knows which tasks it owns

When the system prompt is assembled for a workspace or specialist
agent, `ai_invoke._render_active_continuous_tasks(agent.thread_id)`
reads every `data/continuous/*/state.json` scoped to that thread and
appends a short block:

```
## Active continuous tasks in this workspace

You own the following continuous tasks right now. When the user
discusses any of them — scope, direction, status, the next move —
talk about and MODIFY these in place. Do NOT create a new task
with overlapping scope...

### `zeus-research` — status: awaiting-input
- Objective: Improve deconvolution benchmark...
- Checkpoint policy: `on-milestone`
- Next step #3: benchmark the current stacker on test corpus
- Awaiting question: OK to modify the stacker?
- Plan file: `data/continuous/zeus-research/plan.md`
```

The block is empty (zero overhead) when the workspace has no active
tasks, so the common case is unaffected.

### 3. `[UPDATE_PLAN]` macro

In-place program edits via a new macro:

```
[UPDATE_PLAN name="zeus-research"]
[CONTINUOUS_PROGRAM]
{
  "checkpoint_policy": "on-milestone",
  "success_criteria": [
    "deconvolution runs in under 500ms on the benchmark set",
    "no regression on the test corpus"
  ]
}
[/CONTINUOUS_PROGRAM]
```

Partial payloads are fine — only the fields you provide are merged.
Accepted fields: `objective`, `success_criteria`, `constraints`,
`checkpoint_policy`, `context`, and `plan_text` (a free-form markdown
body written verbatim to `data/continuous/<name>/plan.md`). Unknown
fields are ignored for forward compatibility; invalid values produce a
short rejection line and never touch state.

`UPDATE_PLAN` is workspace-scoped: tasks owned by another thread are
reported as "not found", so one workspace can never silently edit
another's state.

## User-visible behaviour

- If you ask the primary agent to "change the scope" / "update the
  plan" / "adjust the checkpoint policy" / "now the goal is..." of an
  existing task, it emits `[UPDATE_PLAN]` instead of creating a new
  task.
- When you reply to an `awaiting-input` task, the primary agent treats
  your reply as an answer to the task's pending question, not as a
  fresh request. It may emit `[UPDATE_PLAN]` to adjust state and/or
  `[RESUME_TASK]` to continue.
- Configured checkpoint policies are actually respected: `on-milestone`
  no longer stops at arbitrary steps; `on-demand` no longer lets the
  step agent pause on its own whim.

## Compatibility

- **State files**: no migration needed. `checkpoint_policy` already has
  a default at both read and write sites; existing state files without
  the field continue to work with `on-demand` behaviour.
- **Prompt size**: the active-tasks block is tiny (a few lines per
  active task, typical workspaces have 0–3 active tasks). No risk of
  exceeding the 50 000-word system-prompt cap.
- **Non-executive agents**: the `[UPDATE_PLAN]` marker is included in
  `_EXECUTIVE_MARKERS` so collaborative participants cannot emit it.
