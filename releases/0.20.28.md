# Release 0.20.28

**Review-driven hardening**

min_compatible: 0.20.0
breaking: false

## Summary

A deep code review of the project surfaced roughly 100 findings
across runtime, scheduler, collaborative workspaces, process
management, and media. After verification, ~35 confirmed-real items
landed in this release. Several agent-reported "critical" items were
ruled out as false positives during verification and did **not**
drive any code change (e.g. claimed FD leak in `_queue_mutex`,
claimed voice file-handle leak, claimed dataclass lock factory bug).

This is a correctness / security / operability release. Behaviour
for the common case is unchanged; what changes is how the system
fails.

## Critical

- **Auto-updates on Slack / Discord were broken.** The background
  update loop in `bot/bot.py` was calling `check_for_updates()`
  without `await`, so every tick `info` was a coroutine object, the
  `if info:` branch took, and the next subscript raised — or, if the
  harness swallowed it, the loop just no-op'd. One-line fix. This
  path **did** work on the Telegram legacy code path
  (`bot.py:190`, which correctly awaits), so regressions were
  limited to the newer messaging backends.

- **Fire-and-forget `asyncio.create_task(_early_typing())`.** Two
  call sites in `handlers.py` spawned tasks without keeping a
  reference. In practice the runtime would eventually emit
  `Task was destroyed but it is pending!` warnings, and any
  exception inside the coroutine was routed only to asyncio's
  default handler. A new `_spawn_tracked(coro, name=…)` helper
  keeps a strong reference in a module-level set and attaches an
  exception-logging done-callback. Both call sites now use it.

- **Collab store lock was a no-op on Windows.** `fcntl` is absent
  on Windows, so `CollabStore._mutex()` degraded to a thread-lock
  only — multi-instance installs corrupted
  `collaborative_workspaces.json` under concurrent writers. Added
  an `msvcrt.locking(LK_LOCK)` fallback.

- **Race in `collab_bot_added` Flow B.** The provisional workspace
  was published to `CollabStore` **before** the agent was registered,
  and the agent-brief file was written **after**. A message arriving
  in the newly-added group during that window could see the
  workspace but fail to find the agent. Fixed by registering the
  agent first, writing the brief inside a `try/except` with
  `manager.remove_agent(...)` rollback on failure, and only then
  publishing to the routing store.

## Security

- **`[SEND_IMAGE path="..."]` path allowlist.** Agents previously
  could emit an absolute path to any file the bot process could
  read. With prompt injection, an attacker could coax an agent into
  uploading `/etc/passwd`, `~/.ssh/id_rsa`, etc. Paths are now
  validated before any filesystem access: they must resolve under
  the agent's `work_dir`, `data/`, the system tempdir, or (on
  POSIX) `/tmp`. Anything else is refused with a user-visible
  error and a `WARNING` in `bot.log`.

## Concurrency & process lifecycle

- **Agent subprocesses now spawn with `start_new_session=True`
  (POSIX).** Each CLI lives in its own process group, and
  `Agent.interrupt()` signals the whole group via `os.killpg`.
  Previously a CLI that spawned a `node` worker left the
  grandchild behind when its immediate child received SIGTERM —
  re-parented to PID 1 and quietly surviving. Same treatment
  applied to the scheduled-task spawn path in `scheduler.py`.
- **Orphan tracker tree-kill.** `cleanup_on_startup()` now uses
  `os.killpg` on POSIX and `taskkill /T /F` on Windows, and
  re-checks `is_pid_alive` 100 ms after SIGKILL to detect silent
  delivery failures (SELinux, odd permissions).

## Data integrity

- **fsync on queue + state writes.** `_save_queue_unlocked` and
  `continuous.save_state` now `flush() + os.fsync()` before
  `os.replace`. A power loss between write and rename can no
  longer leave a zero-length tmp rotated into place.
- **Snapshot restore validation.** `updater._restore_data_dir`
  rejects tar members that contain absolute paths or `..`
  traversal before calling `extractall`, preventing a malformed
  snapshot from escaping `data/`.
- **Legacy migration revalidation.** `scheduled_at` on migrated
  one-shot entries from `timed_queue.json` is re-validated; corrupt
  legacy entries are skipped with a `WARNING` instead of wedging
  the scheduler.
- **Delivery watcher `returncode` default.** The scheduled-delivery
  watcher now initialises `returncode = 1` before `proc.wait()`, so
  a wait exception no longer hides the exit status.
- **Instruction cache keyed on `(mtime, size)`.** A brief
  deleted-and-recreated fast enough to land on the same mtime is
  now correctly re-read instead of serving stale content.
- **`append_log` is mutex-guarded.** Concurrent threads can no
  longer interleave a single log entry's bytes.

## Collaborative hardening

- **`CollabStore.update_chat_id` guard.** Refuses to promote a
  workspace to `"active"` unless it is currently `"pending"` with
  `chat_id=0`, with an optional `expected_creator_id` check. Tighter
  defence against Flow-A hijack.
- **`_load()` failure is now ERROR, not WARNING.** Collab routing
  is silently degraded if the JSON doesn't parse — operators need
  the louder signal.
- **Unknown-sender default is logged.** Still defaults to
  PARTICIPANT in-memory, but an `INFO` line makes the event
  auditable.
- **Continuous-task reparent warn.** When a continuous task's
  thread-id has no mapped workspace and we reparent to `robyx`,
  the fallback is now a `WARNING` instead of a silent `info`.

## Defaults & knobs

| Setting | Old default | New default | Env var |
|---|---|---|---|
| Stale-claim reset | 300 s | **600 s** | `CLAIM_TIMEOUT_SECONDS` |
| Reminder max age | 24 h | **7 d** | `REMINDER_MAX_AGE_SECONDS` |
| Smoke-test timeout | 60 s (hardcoded) | 60 s (**env-configurable**) | `SMOKE_TEST_TIMEOUT_SECONDS` |
| Voice transcription timeout | 60 s (hardcoded) | 60 s (**env-configurable**) | `VOICE_TIMEOUT_SECONDS` |

Existing `.env` values are preserved — only the defaults moved.

## Tests

1050 passed, 1 skipped. The hardening is additive; no existing
contract was broken.

## Migration

None. `bot/migrations/v0_20_28.py` is a no-op.

## Verification status

The underlying review (`.claude/plans/linear-beaming-steele.md`)
tagged every finding ✅ verified, ⚠️ plausible-but-unverified, or
❌ false positive. Only ✅/⚠️ items with a concrete, testable fix
landed here. Deferred items (needing deeper design work, e.g.
distributed locking for multi-instance setups, pulling
continuous-task branches through the updater, lock-file
process-start-time keying) are flagged in the review and left as
follow-up work.
