# Release 0.22.0

**External collaborative groups — two-way orchestrator↔agent wiring**

min_compatible: 0.21.0
breaking: false

## Summary

Feature 003-external-group-wiring. External collaborative groups
(non-HQ Telegram groups where the owner and collaborators work with a
dedicated Robyx agent) are now fully wired end-to-end:

- The orchestrator can **pre-announce** a group with
  `[COLLAB_ANNOUNCE]` before it exists; when the bot is added to the
  matching group, Flow A binds it and the first in-group message
  references the pre-announced purpose.
- For **ad-hoc** adds (no prior announcement), the first in-group
  message is now a **real AI turn** instead of a byte-identical
  template. The setup agent asks about purpose + inheritance, then
  emits `[COLLAB_SETUP_COMPLETE]` which rewrites the agent brief,
  promotes the workspace `setup → active`, and notifies HQ.
- The orchestrator's system prompt carries a live
  `[AVAILABLE_EXTERNAL_GROUPS]` registry refreshed every turn, so HQ
  always sees what exists without a separate sync step.
- `[COLLAB_SEND]` lets HQ relay a message into a specific active group;
  `[NOTIFY_HQ]` lets a group's executive user surface a short update
  back to HQ's control room.
- The Telegram `my_chat_member` lifecycle is wired: bot removal closes
  the workspace and drops it from the registry; supergroup migration
  rebinds `chat_id` without losing history.
- Unauthorised adders are refused, the bot leaves the chat, and HQ is
  notified (FR-011) — no `CollabWorkspace` is persisted for rejected
  adds.

## Scope

Telegram-only for this iteration. Discord and Slack receive a single
"not yet supported" message on add, and do not auto-leave. This is a
**justified Principle I (Multi-Platform Parity) violation** —
documented in `specs/003-external-group-wiring/plan.md` under
Complexity Tracking. The rationale is captured there in full; the
short version is that Discord guilds and Slack channels differ
materially from Telegram groups in identity, permissions, and invite
semantics, and extending the feature to three platforms in one
iteration would triple scope. A follow-up feature will address the
parity gap once the Telegram implementation is stable.

## Follow-ups (not in this release)
