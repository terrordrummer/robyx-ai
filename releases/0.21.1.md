This is a security + stability point release on top of v0.21.0.

## Fixed

### Security — Discord adapter

- **P2-11 — Voice-attachment downloads are now streamed with a 25 MB
  cap.** `bot/messaging/discord.py` previously loaded the entire
  response body into memory via `resp.read()`. A hostile Discord
  redirect or a crafted attachment URL could point the bot at a
  multi-GB payload and OOM the process. `download_voice()` now uses
  `resp.content.iter_chunked(64 KB)` and enforces
  `_MAX_DISCORD_DOWNLOAD_BYTES = 25 * 1024 * 1024` via two gates:
  a `Content-Length` header short-circuit (rejects the download before
  any bytes are read) and a running-total guard in the streaming loop
  (catches servers that lie about or omit `Content-Length`). The
  partial temp file is unlinked on any failure.

- **P2-12 — The Discord-hostname allow-list is now a module-level
  helper applied to every download path.** Pass 1 S3 added an inline
  HTTPS + hostname check, but it only guarded `download_voice`. Any
  future download path would have silently regressed the SSRF
  protection. Extracted into `_validate_discord_url()`, called before
  every HTTP fetch the adapter performs. The allow-list now covers
  three Discord-hosted domains (and subdomains):
  `discord.com`, `discordapp.com`, `discordapp.net` (the media CDN —
  previously not in the allow-list, could have caused false positives
  on legitimate CDN fetches).

### Stability — Corrupt-file quarantine

- **P2-30 — On-disk state corruption no longer destroys data
  silently.** Closes Pass 1 **F17** (deferred since v0.20.28).
  Before this release, a malformed `data/state.json` or
  `data/collaborative_workspaces.json` was caught by a broad `except`
  that logged a warning and continued with empty in-memory state.
  The very next write (agent registration, collab update) would then
  overwrite the corrupt file, losing the original bytes forever.

  Both `AgentManager._load_state()` and `CollabStore._load()` now
  catch `json.JSONDecodeError` and `UnicodeDecodeError` separately
  and call a new `_quarantine_corrupt_file(path, reason)` helper that
  renames the corrupt file to `<name>.corrupt-<UTC-timestamp>` (e.g.
  `state.json.corrupt-20260416T173805Z`) before falling back to
  empty state. The next save creates a fresh file; the original
  bytes are preserved on disk for post-mortem inspection or manual
  recovery.

  Operators will see a `WARNING` log line identifying the quarantine
  path and reason, and an `ERROR` line explaining how to rebuild
  state through chat.

### Slack adapter — follow-up items closed as "no action"

- **T072** — the Pass 2 plan carried forward two residual Slack
  concerns: event-ID dedup store sizing and bot-token scrubbing in
  error paths. Re-analysis:
  - Event-ID dedup is handled inside `slack-bolt`'s
    `AsyncSocketModeHandler`, not in our code. Not our contract to
    bound.
  - `_bot_token` is only ever used as an `Authorization: Bearer`
    header in `download_voice`; no `log.error` / `log.warning` in
    `slack.py` echoes the token or a URL carrying one.
  - No gap remains. Marking closed.
