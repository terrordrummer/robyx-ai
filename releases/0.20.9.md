## Auto-retry on transient Claude Code stream errors

When the Mac sleeps (or Amphetamine isn't active) the TCP connection behind
the `claude` subprocess can become stale. On wake the CLI emits one of a
small set of transient errors — most visibly `API Error: Stream idle timeout
- partial response received`, but also `connection reset`, `broken pipe`,
`ECONNRESET`, `ETIMEDOUT`. Until now these leaked into chat as raw `AI Error`
strings, even though the correct response is to simply retry the call with a
fresh session id.

### Fixed

- **Stream-retryable detection** (`bot/ai_invoke.py`) — a new
  `STREAM_RETRYABLE_KEYWORDS` list is matched against **all three** delivery
  paths for these errors:
  1. stderr on a non-zero exit (non-streaming backend path).
  2. stderr when streaming returned no `result` event.
  3. the result payload itself — Claude Code frequently exits 0 with the
     error string as its own response.
  A match triggers the same recovery used for session collisions: regenerate
  `agent.session_id`, exponential backoff (up to 16s), retry up to
  `MAX_AI_RETRIES`. The user sees the recovered response instead of an error.

- **Typing-task cancellation** (`bot/handlers.py`) — `_process_and_send` now
  awaits `typing_task` after cancelling it and swallows `CancelledError`,
  closing a narrow race where the cancelled coroutine could surface as an
  unhandled exception.

### Changed

- **Removed duplicate keep-alive** (`bot/ai_invoke.py`) — the per-invocation
  typing loop inside `_invoke_ai_locked` is gone. Since 0.20.6 a continuous
  typing loop in `bot/handlers.py` runs from message receipt until the
  response is delivered, covering both the lock-acquisition window *and* the
  subprocess lifetime. Running two loops in parallel was redundant.

### Files touched

- `bot/ai_invoke.py`
- `bot/handlers.py`
- `tests/test_ai_invoke.py`

### No migration needed

Pure logic changes; no config, schema, or queue format changes.
