# Robyx 0.25.0

Code-review follow-through release. Closes ten real bugs surfaced by a
deep audit of `bot/`, tightens several doc contracts, and adds
thirteen regression tests. No persisted state schema changes — the
`v0_25_0` migration is a no-op bump to keep the version chain
continuous.

## Highlights

Three High-severity fixes that together close two data-loss windows:

- **Legacy migration tracker is now atomic.** `bot/migrations/legacy.py`
  `_save_applied` wrote through `Path.write_text`, leaving a window
  where a SIGKILL or power loss mid-save truncated
  `data/migrations.json`. Next boot treated the truncated file as
  empty and re-ran every legacy migration. Ported the
  `tmp + fsync + os.replace` pattern from `tracker.save` so the
  durability contract is identical for the legacy and chain sections
  of the same file.
- **`v0_23_0` no longer silently drops failing tasks.** If a
  continuous-task migration could not resolve its parent workspace
  (deleted, renamed, or mid-rename at migration time), the old code
  logged at `WARNING` and wrote the done marker anyway — the chain
  advanced and the broken tasks were permanently stranded. The
  migration now raises, the chain halts on that step, and the next
  boot re-runs it; per-task `migrated_v0_23_0` markers ensure
  already-migrated tasks are not touched on retry.
- **Corrupt JSON in `state.json` / `collaborative_workspaces.json` is
  now recovered from the most recent updater snapshot.** Previously
  the load path quarantined the bad file and started with empty
  in-memory state, which the next `save_state` promptly persisted,
  silently erasing the workspace registry / agent fleet. A new
  `_recover_from_snapshot` helper walks
  `data/backups/pre-update-*.tar.gz` newest → oldest, extracts the
  target file at the same relative path, validates it as parseable
  JSON, and atomically installs the first usable copy. Falls back to
  the existing empty-state path only if no snapshot can help; emits
  `CRITICAL` logs in either case so the event is visible.

## What changed — by area

### Data integrity and persistence

- **`bot/memory_store.py::get_connection`** no longer leaks the SQLite
  connection when `PRAGMA journal_mode=WAL` or `executescript` raises.
  Under disk-full / read-only-FS / corrupt-DB conditions the
  connection used to linger until GC; in tight loops
  (`aggregate_active_summaries` across many agents) FDs could pile up.
- **`bot/migrations/legacy.py`** `_save_applied` uses
  `tmp + fsync + os.replace` (see above).
- **`bot/migrations/v0_23_0.py`** halts on partial failure (see above).
- **`bot/agents.py::_recover_from_snapshot`** (new) + wiring in
  `AgentManager._load_state` and `CollabStore._load` (see above).

### Scheduler + continuous tasks

- **`bot/update_plan_macro.py::apply_update_plan_macros`** now clears
  `awaiting_question` and flips status back to `pending` when an
  `UPDATE_PLAN` macro is applied to a task parked in `awaiting-input`.
  Previously the plan was rewritten but the `awaiting-input` status
  and stale question lingered, so the scheduler kept skipping the task
  forever; the user had to issue a separate `RESUME_TASK`.

### Messaging adapters

- **`parse_mode` contract is now documented on `Platform`.** The
  abstract interface spells out that only `"markdown"` is universal;
  other string values are Telegram-specific. `TelegramPlatform.send_to_channel`
  no longer has asymmetric logic versus `send_message` — the new rule
  is `None` or `"markdown"` → Markdown, `""` → opt-out, anything else
  → raw pass-through.
- **`bot/handlers.py::_send_response`** passes
  `platform.max_message_length - 64` to `split_message` instead of
  relying on the Telegram-centric default of 4000. Discord's 2000-char
  ceiling is now honored; long replies no longer silently fail
  `POST /messages`.

### Configuration

- **`bot/config.py::_int_env`** returns `int(raw)` directly. The prior
  `int(raw) or None` silently turned the valid integer `0` into
  `None`, which (while harmless for chat/owner IDs which are never 0)
  contradicted the documented contract.

### Updater

- **`bot/updater.py::_safe_stash_pop`** (new) replaces the eleven
  silent `await _git("stash", "pop", check=False)` call sites. Non-zero
  exit now produces a `WARNING` with explicit recovery instructions,
  so a failed pop after a failed pull does not strand the user's
  local edits unseen.
- **Migration steps tokenize via `shlex.split`.** Prior
  `step.split()` could not survive quoted arguments (e.g.
  `python -m pip install "package[extra]==1.2"` would have been sent
  to `asyncio.create_subprocess_exec` as five tokens including the
  unbalanced quote fragments). Unbalanced quotes now fail the
  migration loud with a clear message.

### Polish

- **`bot/ai_invoke.py`** pre-declares `heartbeat_task` as `None` so
  the finally block uses an explicit `is not None` check instead of
  catching `NameError`.
- **`bot/config.py`** inline comment on `AI_CLI_PATH` points at
  `ai_backend.create_backend` for the lazy detection location.
- **`bot/_bootstrap.py`** docstring no longer mentions the
  unimplemented `PYTEST_CURRENT_TEST` check — the single `.venv/`
  existence check covers both dev and test paths.
- **`bot/topics.py::_sanitize_task_name`** docstring names the
  mapping as non-injective and points at `_validate_new_agent_name`
  as the collision gate.
- **`bot/handlers.py::_strip_executive_markers`** docstring explains
  why `CREATE_CONTINUOUS` / `UPDATE_PLAN` tolerate typographic quotes
  while `CREATE_WORKSPACE` et al. do not (author authority level,
  not an oversight).
- **`bot/bot.py::ensure_single_instance`** Windows fallback branch
  carries an explicit comment about the accepted TOCTOU window and
  the `O_CREAT | O_EXCL` escape hatch.

### Documentation alignment

- **`bot/scheduler.py`** module docstring documents that continuous
  tasks use per-task lock files and orphan recovery instead of the
  claim system.
- **`bot/migrations/runner.py`** break-on-error comment clarifies
  that the tracker returns the last `ok` version, so retry on next
  boot is automatic.
- **`bot/updater.py::apply_update`** docstring lists the seven-step
  safety structure including the snapshot + rollback chain.
- **`bot/collaborative.py::CollabStore._mutex`** and
  **`bot/agents.py::AgentManager._load_state`** document the
  single-process invariant enforced by
  `bot.bot.ensure_single_instance`, so future callers do not silently
  break the assumption.

## Tests

Thirteen new regression tests land with this release; none of the
existing 1684 tests were modified apart from 47 mocks in
`test_handlers.py` that were updated to accept the new keyword
argument on `split_message`. Notable additions:

- `tests/test_config.py` (new file) — six tests for `_int_env`
  including the `"0"` regression.
- `tests/test_migration_v0_23_0.py` — corruption now raises and the
  retry path after workspace recovery is exercised end to end.
- `tests/test_telegram_platform.py::TestSendToChannelParseMode` — four
  tests pinning down the new `parse_mode` contract (default, explicit
  markdown, empty-string opt-out, Telegram-native HTML passthrough).
- `tests/test_handlers.py::test_split_message_receives_platform_max_length`
  — regression asserting Discord's 2000-char cap flows through.
- `tests/test_update_plan_macro.py` — `UPDATE_PLAN` clears
  `awaiting-input` + does not touch `running` tasks.
- `tests/test_updater.py` — `shlex` tokenization of quoted arguments
  and unbalanced-quote detection.
- `tests/test_agents.py` — two recovery tests (corrupt JSON recovers
  from newest snapshot; recovery walks past an unusable newest
  snapshot to an older good one).
- `tests/test_collaborative.py` — `CollabStore` recovery equivalent.

Full suite: **1697 passed, 1 skipped**.

## Upgrade notes

- Operators on 0.24.3 upgrade to 0.25.0 via the built-in auto-updater
  or `git pull` + `systemctl restart robyx`. The migration framework
  runs `v0_25_0.upgrade` as a no-op; no operator action required.
- If a prior release landed a corrupt `state.json` or
  `collaborative_workspaces.json` on disk, the next boot will
  auto-recover from the most recent updater snapshot and log at
  `CRITICAL` — read the log and confirm the recovered state is
  acceptable before resuming normal operation.
- If a previous v0_23_0 run stranded tasks by writing the done marker
  despite errors, they remain stranded. This release does not retry
  them automatically (the done marker is still honored). Remove
  `data/migrations/v0_23_0.done` manually if you want the new, stricter
  migration to re-process those tasks.
