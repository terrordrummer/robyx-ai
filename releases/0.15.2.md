## The reminder skill actually works now

A focused fix release that closes the silent regression that made the v0.15.0 reminder skill ineffective on every existing fleet — including the very Mac install that prompted v0.15.0 in the first place. The bug, root cause, and fix are documented in detail below because this is the second time we've shipped a "fix" for the same user-visible symptom and it deserves to be called out clearly.

This release also fixes a latent `NameError` in the Telegram boot path that's been silently swallowed since v0.14.0, and removes one leaked personal task entry that was breaking the linknx clone.

---

## Why this release exists

### The user-visible failure

The user said: "L'agente assistente non mi ha ricordato che sono figo all'ora che ho stabilito" — the assistant didn't fire the reminder. After v0.15.0 and v0.15.1 both shipped specifically to fix this. Investigation against the runtime install on the user's Mac confirmed:

1. `data/migrations.json` listed `0.15.0-reset-sessions-for-reminder-skill: success`.
2. `data/state.json` listed `assistant.session_id = b2c3d4e5-f6a7-8901-bcde-f12345678901` — the **same** hardcoded fake UUID from before v0.15.0, and `message_count: 7` (so the agent had been used 7 times since the migration "ran").
3. `data/reminders.json` did not exist.
4. `bot.log` had **zero** occurrences of `REMIND queued` or `Reminder rejected` despite the user having sent multiple "ricordami…" messages to the assistant. The assistant never emitted `[REMIND ...]` because it was running with the pre-v0.15 system prompt that doesn't know about the pattern.

### The root cause

The v0.15.0 migration mutated `data/state.json` directly:

```python
async def _reset_sessions_for_reminder_skill(platform) -> bool:
    state = json.loads(STATE_FILE.read_text())
    for name, agent_data in state["agents"].items():
        agent_data["session_id"] = str(uuid.uuid4())
        ...
    STATE_FILE.write_text(json.dumps(state, indent=2))
```

But by the time the migration ran, the `AgentManager` had **already loaded `state.json` into memory** with the old session_ids:

```
main()                              # BOOT TIME
  ├── manager = AgentManager()      # loads state.json: assistant.session_id = b2c3d4e5-...
  └── _run_telegram(...)            # schedules boot_notify for +3s

(3 seconds later)
boot_notify()
  └── run_pending_migrations()      # rewrites state.json: assistant.session_id = fresh-uuid-A
                                    # but manager.agents["assistant"].session_id is still b2c3d4e5-...

(user sends a message)
ai_invoke._invoke_ai_locked(agent, ...)
  ├── claude --resume b2c3d4e5-...                # uses in-memory session_id (the OLD one)
  ├── ...                                          # successful turn
  └── manager.save_state()                         # writes ALL agents from memory back to disk
                                                   # state.json["assistant"]["session_id"] is now b2c3d4e5-... AGAIN
```

The migration's mutation was clobbered by the AgentManager on the very next interaction. The migration tracker still said `success` because the migration *function* returned True. But the agent ran with the old prompt forever.

### The v0.15.1 updater path had the same bug

v0.15.1 promoted the v0.15.0 one-shot migration into a permanent updater contract: `apply_update` would diff `<pre>..HEAD` and call `invalidate_sessions_for_paths(STATE_FILE, changed_paths)` for any release that modified a system prompt or an agent brief. Same fundamental flaw: the file mutation was clobbered by the running bot's next `save_state()` call. v0.15.1 tests passed because they only checked the file *immediately after* the call — never simulated a downstream interaction.

### Why the v0.15.0 / v0.15.1 tests didn't catch it

Both releases tested against a `state.json` file as a dict, never against a real `AgentManager`. The tests verified "the file on disk has the expected content right after the call", which is true. They didn't verify "the in-memory state and the on-disk state stay in sync after a downstream save_state". v0.15.2 ships that test explicitly: `tests/test_agents.py::TestResetSessionsSurvivesSubsequentSaveState::test_direct_state_json_mutation_would_be_clobbered` **demonstrates** the bug pattern with a real `AgentManager` and asserts that direct file mutations are lost. If the bug ever comes back, this test will scream.

## The fix

### `AgentManager.reset_sessions(agent_names)` is the only correct path

New public method on `AgentManager`. Mutates `self.agents` in place using the same convention as the existing placeholder-UUID sanitiser (`uuid.uuid4()`, `session_started=False`, `message_count=0`), then immediately calls `self.save_state()`. The in-memory and on-disk copies are always in sync after the call. Every other field of every agent — `thread_id`, `name`, `work_dir`, `model`, `description`, `created_at`, `last_used`, `busy` — is left verbatim.

```python
def reset_sessions(self, agent_names: set[str] | None = None) -> list[str]:
    target = list(self.agents.keys()) if agent_names is None else [
        n for n in agent_names if n in self.agents
    ]
    for name in target:
        agent = self.agents[name]
        agent.session_id = str(uuid.uuid4())
        agent.session_started = False
        agent.message_count = 0
    self.save_state()
    return sorted(target)
```

`agent_names=None` means "all known agents". An explicit set names targets and silently ignores unknown names — that protects renames and removals when the diff-driven updater is the caller.

### Migrations and the updater both go through the manager

The migration framework now passes the live `AgentManager` to every migration:

```python
@migration(id="...", description="...")
async def my_migration(platform, manager):
    ...
    manager.reset_sessions(None)  # the only correct path
```

Channel-rename migrations accept `manager` and ignore it — they only touch the platform. State-mutating migrations call `manager.reset_sessions(...)` and never read or write `state.json` directly.

The updater takes a new `manager` keyword argument:

```python
success, msg = await apply_update(version, notify_fn=..., manager=manager)
```

Inside `apply_update`, the diff-driven invalidation step calls `invalidate_sessions_via_manager(manager, changed_paths)` (the new entry point in `bot/session_lifecycle.py`) which routes the actual reset through `manager.reset_sessions(...)`. If `manager=None` is passed (legacy callers, CLI invocations outside the bot process), the invalidation step is skipped with a warning and the update still completes.

### A new migration runs on existing fleets

Because v0.15.0's migration is already marked `success` in every existing tracker, the migration framework will not re-run it. v0.15.2 ships a **new** migration with a fresh ID:

```python
@migration(id="0.15.2-reset-sessions-after-clobber-fix", ...)
async def _reset_sessions_after_clobber_fix(platform, manager):
    if manager is None:
        return True
    reset = manager.reset_sessions(None)
    ...
    return True
```

On the first boot after upgrading to v0.15.2, this migration runs (the ID is not in `migrations.json` yet), goes through the manager, and the reset actually persists. Fresh installs that have never seen v0.15.0 will run both migrations in order — the first via the new code path, the second as a redundant no-op (everything is already a fresh session).

### Bug fix: `manager` undefined in `boot_notify`

Latent since v0.14.0 — `bot/bot.py:441` referenced `manager` from a scope where it was not defined, raising `NameError` on every boot. The exception was caught by the surrounding `try`, the `heal_detached_workspaces` step never ran, and the only trace was a stack trace in `bot.log`. The user's runtime install shows it firing every reboot since v0.14: `2026-04-08 23:03:40`, `23:10:40`, `23:25:07`...

v0.15.2 threads the `AgentManager` through `_run_telegram` / `_run_discord` / `_run_slack` and captures it in the `boot_notify` closure via the job data:

```python
app.job_queue.run_once(
    boot_notify, when=3,
    data={"platform": plat, "manager": manager},
)
```

Inside `boot_notify`, `boot_manager = context.job.data["manager"]` is now the live AgentManager. `heal_detached_workspaces(boot_manager, ...)` works again, and `run_pending_migrations(boot_plat, boot_manager)` can pass the manager through to state-mutating migrations.

### Bug fix: leaked `zeus-engine` row in `tasks.md`

The `zeus-engine | agents/zeus-engine.md | scheduled | hourly | yes | opus | 726 | ...` row in `tasks.md` was leaked into the GitHub repo from the user's *working* directory. The agent file `agents/zeus-engine.md` only exists in the user's `~/Workspace/products/kael-ops/agents/` (working dir), not in the runtime install nor in any clone. Two visible failures:

- On `linknx` (a fresh clone of the repo): scheduler errored on every cycle because the agent file did not exist.
- On the user's Mac runtime install: the same `ERROR: Agent file not found: agents/zeus-engine.md` flooded `bot.log` every 10 minutes.

v0.15.2 surgically removes the row. The other personal entries in `tasks.md` and `specialists.md` (`assistant-check`, `system-monitor` thread_id, `deploy` specialist, the tracked personal agent files in `agents/`) are intentionally **out of scope** for this release. Cleaning them up requires untracking those files and shipping a backup migration in `apply_update` so the user's local data is not destroyed by the next pull. That's the v0.16 release.

## Tests

**808 passing** (from 803 baseline). The net is small because much of v0.15.2 is rewriting v0.15.0 / v0.15.1 tests that were structurally wrong. The important additions:

- `TestResetSessions` and `TestResetSessionsSurvivesSubsequentSaveState` in `tests/test_agents.py` — including `test_direct_state_json_mutation_would_be_clobbered` which **demonstrates** the v0.15.0 bug pattern as a regression guard. If anyone ever rewrites `save_state` semantics, this test will catch it before the bug ships again.
- `TestResetSessionsAfterClobberFix` in `tests/test_migrations.py` for the new v0.15.2 migration, including a registration-order assertion.
- `TestRunPending::test_manager_is_passed_to_each_migration` — guard against future refactors that drop `manager` from the migration runner.
- `TestInvalidateSessionsViaManager` in `tests/test_session_lifecycle.py` — uses a fake `AgentManager` that records every `reset_sessions` call so the assertions are about *what the manager was asked to do*, not about a post-hoc file read.

## Compatibility

- **No breaking changes for users.** The auto-updater applies v0.15.2 like any other release. On first boot, the new migration regenerates the assistant's session via the manager-aware path; the next message creates a fresh Claude session that bakes in the v0.15 system prompt with `[REMIND]`; reminders work end-to-end from there.
- **Developer-facing: `Migration.apply` signature changed** from `(platform)` to `(platform, manager)`. Anyone with out-of-tree migrations needs to add a second `manager` parameter. Channel-rename migrations can ignore it.
- **Developer-facing: `apply_update` accepts a new `manager=` keyword argument.** Calling `apply_update(version)` without it still works — invalidation is skipped with a warning. The bot's auto-update path now always passes it.
- **No new dependencies.**
- **No `requires_migration: true`** in the frontmatter — no shell migration steps to run for this release.
