# Robyx 0.24.2

Bugfix release: the fire-and-forget invariant of continuous tasks is now
actually enforced. Tasks that used to stop after the first step and
refuse to resume are handled end-to-end.

## The bug

After starting a continuous task the user would see:

1. the first step run,
2. the task park in `awaiting-input`,
3. a chat message reading "stopped by user" (even though the user had
   not asked to stop anything),
4. any attempt to resume via the primary agent silently fail — the
   task stayed stuck forever.

Three independent defects reinforced each other:

1. **Broken resume contract.** `bot/ai_invoke.py` told the workspace
   agent to emit `[RESUME_TASK]` for tasks in `awaiting-input`, but
   `bot/lifecycle_macros.py` rejected that state — only `paused` and
   `rate-limited` were on the resume whitelist. Every resume attempt
   returned "Task X non è in pausa" with no state change.
2. **Scheduler stalemate.** `_handle_continuous_entries` skipped every
   entry whose state was `awaiting-input`, unconditionally — regardless
   of the task's declared `checkpoint_policy`. A step agent drifting off
   policy (or running against a stale prompt template) could park the
   task once and never come back.
3. **Ambiguous stop provenance.** `_stop_task` / `_pause_task` always
   recorded `canceled_reason = "stopped by user"` even when the macro
   came from the agent's own initiative, making it impossible after the
   fact to tell a user-driven stop from an agent-driven one.

## What changed

### Fire-and-forget invariant (explicit)

`templates/prompt_workspace_agent.md` now states the invariant directly:
**once a continuous task is running, user chat messages do NOT touch
it** unless the user explicitly asks, or the step agent itself needs
input within the agreed checkpoint policy. This section is always
present in the system prompt, not conditional on the "Active continuous
tasks" block.

### Confirmation gate for STOP / PAUSE

A new section in the workspace-agent prompt mandates a **two-turn
confirmation gate** before emitting `[STOP_TASK]` or `[PAUSE_TASK]`:

1. recognise the user's intent from the whole conversation (reasoning,
   not keyword matching);
2. identify the target (ask which task if ambiguous);
3. ask "do you confirm?" in a dedicated turn — no macro yet;
4. only on an unambiguous affirmative reply emit the macro.

Status queries, scope changes, frustration with a step, answering a
step agent's question, and the agent's own "this task doesn't look
productive" judgement are **explicitly** listed as NOT grounds to
stop. `[RESUME_TASK]` does not require confirmation — resuming is
aligned with the invariant.

### Resume whitelist extended

`bot/lifecycle_macros.py::_resume_task` now accepts `awaiting-input`
alongside `paused` and `rate-limited`. `bot/continuous.py::resume_task`
clears any stale `awaiting_question` on resume so the next scheduler
tick sees a clean `pending` state.

### Server-side `on-demand` enforcement

`bot/scheduler.py` now auto-demotes any task parked in `awaiting-input`
with `checkpoint_policy == "on-demand"` back to `pending`, logging a
policy-violation warning. The step template has always forbidden this
transition under `on-demand`; the scheduler is now a belt-and-suspenders
enforcer in case a step agent drifts. Tasks under `on-uncertainty`,
`on-milestone`, or `every-N-steps` are left alone.

### Stop/pause reason attribution

`DispatchContext` grows an optional `user_message` field. `bot/handlers.py`
propagates the originating user message into the lifecycle dispatch
context, and `_stop_task` / `_pause_task` now build reason strings of
the form `"stopped by user: <user-message-snippet>"`. Audits can now
distinguish a user-driven stop from an agent-driven one.

### Tests

- `tests/test_lifecycle_macros.py::test_resume_from_awaiting_input_clears_question`
  pins down the new resume contract.
- `tests/test_lifecycle_macros.py::test_stop_reason_includes_user_message_snippet`
  exercises the reason-enrichment path.
- `tests/test_scheduler.py::TestOnDemandAutoDemote` covers the
  auto-demote helper across all four checkpoint policies and
  non-awaiting states.

### No functional change outside continuous tasks

- No schema change, no state migration logic. `bot/migrations/v0_24_2.py`
  is a no-op and exists only to keep the version chain continuous.
- Periodic, one-shot, and reminder queue entries are unaffected.

## Upgrade notes

- Operators on 0.24.1 upgrade to 0.24.2 via the built-in auto-updater or
  `git pull` + `systemctl restart robyx`. No operator action is
  required.
- Already-stuck continuous tasks recover automatically: on the next
  scheduler tick the auto-demote helper moves them from
  `awaiting-input` (under `on-demand`) back to `pending`, and the loop
  resumes. If a task really does need user input, its policy should be
  changed via `[UPDATE_PLAN]` to `on-uncertainty` / `on-milestone` /
  `every-N-steps` before resuming.
