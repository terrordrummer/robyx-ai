## Event Loop Responsiveness Fix

### Fixed: Synchronous subprocess calls blocking message processing

All `subprocess.run()` calls that ran inside the asyncio event loop have been
converted to `asyncio.create_subprocess_exec()`. These blocking calls could
freeze the entire bot for up to 9 minutes, preventing any Telegram message
from being received or processed.

**Affected modules:**
- **`bot/updater.py`** — `_git()` and all callers (`fetch_remote_tags`,
  `check_for_updates`, `apply_update`). The `git fetch --tags --force`
  network call was the primary culprit.
- **`bot/scheduler.py`** — `check_lock()` and `get_running_tasks()` now
  async; process-check calls no longer block the scheduler cycle.
- **`bot/topics.py`** — `_setup_git_branch()` git operations now async.
- **`bot/process.py`** — `get_process_name()` and `is_ai_process()` now
  async. Added `_sync` variants for pre-event-loop startup code.

### Added: Continuous typing indicator

- **`bot/handlers.py`** — A persistent typing loop now fires every 4 seconds
  from the moment a message is received until the response is fully sent.
  Previously the typing indicator only started after lock acquisition and
  prompt construction, leaving the user with no feedback for several seconds.
