## Correctness & robustness pass (P0 + P1)

This release lands ten targeted fixes from the architecture review
documented in the session plan file. Two critical concurrency races,
one undocumented deployment assumption, and seven robustness gaps are
closed. No feature changes, no schema changes, no migration required.

### P0 — correctness

- **`bot/scheduler.py`** — Continuous task dispatch now persists
  `state=running` **before** writing the lock file. The prior order
  allowed a crash between the two writes to leak a dead-PID lock
  while `state` stayed pre-running, which hid the orphan from the
  existing recovery branch (`check_lock` → `mark_step_failed`) and
  let the next scheduler cycle silently re-dispatch, overwriting
  `output.log` and stomping on the prior attempt.

- **`bot/scheduler.py`** — Stale-claim reconciliation in
  `_reconcile_task_results()` now distinguishes three outcomes:
  entry fully removed (INFO), claim token mismatch while status is
  not dispatched (WARNING), and — most importantly — dispatched task
  whose claim cannot be recorded (ERROR, because the next cycle may
  run the same task again). The previous single WARNING hid the
  duplicate-dispatch risk.

- **`bot/scheduler.py`** — Added `_queue_mutex()` context manager
  combining the existing `threading.Lock` with a POSIX
  `fcntl.LOCK_EX` on a sidecar `data/queue.json.lock`. Two bot
  processes (e.g. during a rolling restart) can no longer
  double-claim the same queue entry. No-op on non-POSIX systems.
  All eight queue critical sections migrated to `_queue_mutex()`.

- **`bot/agents.py`** — `AgentManager.save_state()` now writes
  `state.json` atomically via `tmp + os.replace`, matching the
  pattern already used by `save_queue()` and `continuous.save_state()`
  everywhere else in the codebase.

- **`bot/config.py`, `bot/bot.py`** — `CHAT_ID` / `OWNER_ID` default
  to `None` instead of crashing `int(None)` at module-import time;
  `bot.main()` raises a clear `RuntimeError` only when
  `PLATFORM=telegram` actually needs them. Unblocks test collection
  outside the conftest fixture.

### P1 — robustness

- **`bot/scheduler.py`** — Reminder max-age guard: reminders older
  than `REMINDER_MAX_AGE_SECONDS` (default 24h, in `bot/config.py`)
  are marked failed with `failure_reason="expired"` instead of
  retrying forever until `MAX_REMINDER_ATTEMPTS` exhausts. Prevents
  queue bloat when a reminder keeps hitting a transient error. The
  existing max-attempts path now also annotates `failure_reason`
  for observability.

- **`bot/scheduler.py`** — `cleanup_stale_locks_on_startup()` runs
  on the first scheduler cycle and removes every `data/*/lock`
  whose PID is dead or recycled into a non-AI process. The existing
  `check_lock()` only cleaned lazily during task polling, so locks
  on workspaces with no queue entry lingered indefinitely.

- **`bot/orphan_tracker.py`** (new) — Small JSON registry
  (`data/active-pids.json`) of subprocesses the bot believes are
  alive. `register()` / `unregister()` wrap every `invoke_ai`
  spawn. `cleanup_on_startup()` is called from `bot.main()` before
  backend init and force-kills survivors — but only if the PID is
  both alive and mapped to one of our process names (claude, codex,
  opencode, python, node). Recycled PIDs are left alone.

- **`bot/messaging/base.py`** — New `retry_send()` helper with
  exponential backoff (3 attempts, 1-2-4s). Transient platform
  hiccups (network blips, 5xx responses) no longer lose a message on
  the first try. Applied to `send_message` in all three adapters
  (`telegram.py`, `discord.py`, `slack.py`).

- **`bot/ai_invoke.py`** — Heartbeat watchdog in `_invoke_ai_locked`:
  an asyncio side-task logs `"agent X still running (Ys elapsed)"`
  every 60s while the subprocess runs. Operators scrolling `bot.log`
  can now tell a genuinely long task from a hung one. User-facing
  liveness is already covered by the typing-indicator loop in
  `handlers.py` — no additional user-visible noise.

### Tests

971 passed, 1 skipped, 3 warnings. No regressions. The new behaviour
is exercised through the existing scheduler, agents, and adapter
test suites (stale claim, file-lock semantics, atomic write, retry
logic).

### No migration needed

All changes are backward-compatible:
- New queue field `failure_reason` is optional and only added on
  transition to `status=failed`.
- The `data/queue.json.lock` sidecar is created on first mutation
  and need not exist on upgrade.
- `data/active-pids.json` is created lazily on first subprocess
  spawn.

The bootstrap migration `v0_20_17.py` is therefore a no-op — the
chain-continuity contract still requires the module to exist so the
CI test `TestChainContract` keeps passing.

### Relation to the review plan

This release covers the first two buckets (P0, P1) of the
architecture review plan. P2 (simplification — unifying task types,
extracting system prompts, single-pass pattern extraction) and P3
