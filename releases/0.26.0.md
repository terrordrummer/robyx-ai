# Robyx 0.26.0

**Continuous-task observability & lifecycle robustness.** Ships spec 006,
a coordinated fix for nine interconnected fragilities surfaced by real
incidents in zeus-rd-172 (17 silent iterations with scheduler warning
spam), zeus-research (lost in HQ noise, then `name_taken` on recreate),
zeus-engine (stale lock blocked task for 8 days), align-research
(awaiting-input stopped the world without a visible signal).

## What users will notice

1. **Each continuous task now owns a dedicated topic.** Created at
   `[CREATE_CONTINUOUS]` time as `[Continuous] <name> · ▶`. Step
   deliveries, state markers, pinned questions, and final notices go
   there. Parent workspace chats stay clean.
2. **Zero scheduler-driven notifications in HQ.** Dispatches,
   completions, lock recoveries, orphan detections, state transitions
   are journaled — not pushed. Ask the orchestrator
   *"what happened in the last N hours?"* and it will emit
   `[GET_EVENTS since="Nh"]` to produce a chronological summary.
3. **Awaiting-input is unmistakable.** On transition, the delivery is
   pinned in the task's topic and the topic title gets the `· ⏸`
   suffix. After 24h of continued silence, exactly one reminder is
   posted in the same topic.
4. **Lifecycle is four distinct operations.**
   - `stop` — pause dispatches, topic + history preserved, resumable.
   - `resume` — back to pending from `stopped / awaiting_input / rate_limited / error`.
   - `complete` — terminal success; state + topic preserved as record.
   - `delete` — archives topic (`[Archived] <name>`), purges state &
     agent file, **frees the name for immediate reuse**. This closes
     the `name_taken` trap zeus-research hit.
5. **Autonomous recovery from subprocess death.** Lock file carries a
   heartbeat line refreshed every 30s by the parent bot's watchdog.
   If the heartbeat goes stale (SIGKILL, OOM, host reboot), the
   scheduler reclaims the lock on its next cycle — no bot restart
   needed. Zeus-engine would have unblocked itself, not waited 8
   days.
6. **Orphan spam eliminated.** Three consecutive orphan detections
   escalate to exactly one `orphan_incident` event with a diagnostic
   payload and transition the task to `error`. Zero further warnings
   until the task is reset or recreated.
7. **Workspace-close no longer loses in-flight output.** The new
   drain path waits up to `drain_timeout_seconds` (default 60 min,
   per-task overridable via `[CONTINUOUS_PROGRAM]`'s
   `drain_timeout_seconds` field, range `[60, 21600]`) for the
   running step to finish, then delivers with a `⚠ workspace closed`
   header.

## Config knobs (env vars with sane defaults)

| Var | Default | Purpose |
|---|---|---|
| `LOCK_HEARTBEAT_INTERVAL_SECONDS` | 30 | Watchdog refresh cadence |
| `LOCK_STALE_THRESHOLD_SECONDS` | 300 | Heartbeat age beyond which the lock is stale |
| `ORPHAN_INCIDENT_THRESHOLD` | 3 | Consecutive detections before escalation |
| `EVENT_RETENTION_DAYS` | 7 | Journal shard retention |
| `EVENT_MAX_HOT_BYTES` | 10 MB | Size-based rotation threshold |
| `AWAITING_REMINDER_SECONDS` | 86400 | 24h window to fire the reminder |
| `DRAIN_TIMEOUT_DEFAULT_SECONDS` | 3600 | Default drain-on-close window |
| `TOPIC_UNREACHABLE_RETRY_WINDOW_SECONDS` | 300 | Silent recreation retry budget |

## Migration `v0_26_0`

Idempotent forward-only. Per-task timestamp gate + process-wide done
marker. Partial failures are re-entrant — only the unfinished tasks
are retried on the next run.

1. Creates `data/events/` directory and `data/events.jsonl`.
2. For each existing continuous task (excluding `deleted`):
   - opens a `[Continuous] <display_name>` topic,
   - persists `dedicated_thread_id` + `drain_timeout_seconds` default
     in state,
   - re-points the queue entry,
   - applies the state-marker title suffix,
   - seeds one `migration` event in the journal with old / new
     thread_ids.
3. If the task is `awaiting_input` with a stored question, posts and
   pins a retroactive awaiting message so the user lands on the
   correct state.

Rollback to `v0.25.x` is safe: older code ignores the new state
fields and tolerates the dedicated topic as a stray channel.

## Known limitations

- **Slack UX asymmetry (Constitution Principle I, documented).** Pins
  are workspace-wide, not topic-scoped; `conversations.archive` is
  permanent (admin required to unarchive). Adapter logs one WARN per
  session per affected method.
- **FR-002a end-to-end HQ last-resort push** not yet wired end-to-end.
  Infrastructure in place (`check_lock_status`,
  `topic_unreachable_since_ts`, `hq_fallback_sent`); final push on
  unreachable topic + user-actionable event is a follow-up PR.
- **`/stop /resume /complete /delete` slash commands** not yet wired;
  the macro path (`[STOP_TASK]`, `[RESUME_TASK]`, `[COMPLETE_TASK]`,
  `[DELETE_TASK]`) is fully functional.
