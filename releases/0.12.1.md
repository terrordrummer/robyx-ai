## What's New

### The main channel is now the "Command Bridge"

KaelOps's control channel finally has an identity. What used to be "General" (Telegram's default) or a nondescript control room is now the **Command Bridge** ŌĆö the single point where Kael, the Principal Orchestrator, lives and from where the entire agent fleet is coordinated.

Three things change with this release:

1. **The platform's main channel is automatically renamed on first boot.** Telegram's General topic, Discord's control channel, and Slack's control-room channel are all renamed to "Command Bridge" (or the slug `command-bridge` where platform rules require lowercase/no-space names). This happens via the new migration framework (see below), runs exactly once, and is skipped on subsequent boots.

2. **Kael's system prompt now documents the Command Bridge as a *coordination-only* space.** The old prompt said "You live in the main channel" with a flat list of responsibilities. The new prompt introduces an explicit "what belongs on the Bridge" vs "what does NOT belong on the Bridge" contract: Kael handles fleet status, workspace creation, delegation, and meta-operations, but does **not** run project-specific work like R&D iterations, builds, or deploys directly from the Bridge. When a request implies real work inside a project, Kael offers delegation via `[DELEGATE @agent: ...]` or suggests switching to the workspace topic.

3. **User-visible strings updated** in `bot/i18n.py` ŌĆö help text, `no_workspaces`, `unmapped_topic`, `focus_off`, and the boot notification now reference the Command Bridge explicitly. Boot message is now `KaelOps vX.Y.Z ŌĆö Command Bridge online.`

### Migration framework (`bot/migrations.py`)

This release introduces a small but load-bearing piece of infrastructure: a **migration framework** that runs post-update instructions automatically on the next boot after an update is applied. Each migration is a registered async function that runs exactly once per deployment and can touch the filesystem, the platform API, or anything else the new version needs.

**Design**:

- Migrations are declared with `@migration(id=..., description=...)` in `bot/migrations.py`. Order is source order.
- The tracker file `data/migrations.json` records every attempted migration with its outcome (`success` / `failed` / `error`). A migration is **never retried** once attempted ŌĆö this is intentional, so a migration that cannot succeed (missing permissions, missing platform feature) does not block every subsequent boot.
- Migrations must be idempotent where possible ŌĆö check current state and return `True` as a no-op if the work is already done.
- Exceptions are caught, logged with full traceback, and recorded as `error`. The bot continues booting.
- The runner is invoked from each platform's startup hook: Telegram in `boot_notify`, Discord in `on_ready`, Slack in `_run`. It runs **before** the boot notification, and any migrations executed this boot are summarised inline in the boot message.

**Why it matters**: previously, every version that needed post-upgrade work had to either ship as a "manual intervention" release (bumping `min_compatible`) or depend on ad-hoc patching scripts. The framework makes this routine ŌĆö any future cleanup, channel rename, state normalisation, or platform-side resource provisioning becomes a 5-line migration.

### New Platform method: `rename_main_channel(display_name, slug)`

Added to the `Platform` ABC and implemented in all three adapters:

- **Telegram**: calls `Bot.edit_general_forum_topic(chat_id, name)` with the display name (Telegram supports spaces). Requires the bot to have the `can_manage_topics` admin right.
- **Discord**: calls `channel.edit(name=slug)` on the configured control channel. Requires `manage_channels`. Idempotent: checks `channel.name` first and returns `True` without an API call if already renamed.
- **Slack**: calls `conversations_rename(channel, name=slug)` after an idempotency check via `conversations_info`. Requires `channels:manage`.

The method returns `bool` rather than raising. Failures are logged in the adapter; the migration runner records them as `failed` and moves on.

## Operator Notes

- **Permission check before upgrading**: for the Command Bridge rename to succeed, your bot needs the rename permission on its main channel:
  - **Telegram**: bot must be admin with "Manage Topics" enabled
  - **Discord**: bot must have `Manage Channels` on the control channel
  - **Slack**: bot token must have `channels:manage` (or `groups:write` for private channels)
  
  If the permission is missing, the migration logs an error, records itself as `failed`, and the boot continues normally. You can grant the permission manually and run the rename yourself from the platform UI ŌĆö the migration will never retry, so there is no loop.

- **Migrations tracker file**: `data/migrations.json`. Inspect it with `cat data/migrations.json | python -m json.tool`. To force a migration to re-run, delete its key from the file and reboot the bot.

Drop-in upgrade from v0.12.0 ŌĆö pull and restart.
