## Updater hotfix — smoke test + detached-HEAD recovery

Two long-standing updater bugs surfaced when rolling out v0.20.20 on
real installations. Both were latent from much earlier releases but had
been masked by users performing manual `git pull` updates; the first
attempt to auto-update the Mac and Linux deployments in this cycle hit
them head-on. This release is a pure updater hotfix — no behavioural
changes to the bot itself.

### Fixed — smoke test invocation (Mac / any POSIX install)

**Bug**: `_post_update_smoke_test()` ran
`python -c "import bot.bot"` with `cwd=PROJECT_ROOT`. Inside
`bot/bot.py:16` the very first statement is `import _bootstrap`, which
searches `sys.path` — and `sys.path` contained `PROJECT_ROOT` but not
`PROJECT_ROOT/bot`, so the module was never findable. Every auto-update
after v0.20.14 (when the smoke test was introduced) aborted with
`ModuleNotFoundError: No module named '_bootstrap'` and rolled back.
Users who upgraded via auto-update never noticed because the rollback
did not surface the real import chain; those who upgraded manually via
`git pull` bypassed the smoke test entirely.

**Why the "obvious" fixes don't work**: adding `bot/` to `PYTHONPATH`
makes Python resolve `import bot` to `bot/bot.py` (the module file)
instead of `bot/` (the package directory), because a module shadows a
namespace package when both are on `sys.path`. So simply injecting the
path is not an option.

**Fix**:
- **`bot/bot.py`** — new `--smoke-test` flag. When present, after all
  module-level imports have completed (the production boot path up to
  but not including `main()`), the process exits 0. No sockets opened,
  no pid lock acquired, no network I/O.
- **`bot/updater.py`** — `_post_update_smoke_test()` now invokes
  `python bot/bot.py --smoke-test`, exactly the same command shape the
  production service uses (`python bot/bot.py`). `sys.path[0]` ends up
  being `bot/`, `import _bootstrap` resolves, and the test verifies
  every transitive import the real startup would hit.

### Fixed — detached-HEAD pull recovery (Linux / any install that rolled back)

**Bug**: the rollback path does `git checkout v<old_version>`, leaving
HEAD detached at the old tag. From that state `git pull --ff-only`
aborts with `fatal: You are not currently on a branch`, so *every
subsequent update attempt* fails — the install is stuck at whatever
version triggered the original rollback, with no way to recover via
auto-update. The Linux deployment hit this after an earlier rollback to
v0.20.15.

**Fix**: `apply_update()` now checks HEAD attachment with
`git symbolic-ref --quiet HEAD` right after the stash step. If HEAD is
detached, it runs `git checkout main` before the pull. If the checkout
fails the update aborts cleanly with `could not reattach to main: ...`
(so operators are not silently stuck). Local changes were already
stashed one step earlier, so there is nothing to lose.

### Migration

None. `bot/migrations/v0_20_21.py` is a no-op.

### Recovery for affected installs

Because the broken v0.20.19/0.20.20 updater runs *in-process* during
the upgrade, an affected install cannot auto-update its way out: the
smoke test fix only takes effect once the new code is already on disk
and running. One-shot manual recovery on each machine:

```bash
cd <project_root>
git checkout main        # only needed on Linux or any detached install
git pull --ff-only
./install/install-*.sh   # or restart the service
```

From v0.20.21 onwards auto-update works normally.
