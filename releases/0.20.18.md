## P2 simplification pass — smaller config, shared spawn helper

Follow-up to v0.20.17, landing the simplification bucket (P2) of the
architecture review. Pure refactoring: no behaviour change, no schema
change, no feature addition. The goal is to pay down three clearly
costly spots so future work is easier.

### Changed

- **`bot/config.py`** — dropped from 604 to 233 lines by lifting the
  three in-file system prompts (orchestrator, workspace agent,
  focused agent) into standalone Markdown files under `templates/`.
  A new `_load_prompt()` helper reads them at import time. The
  prompts are now diffable, editable, and version-controllable
  without touching Python, which makes the A/B iteration loop on
  prompt text one commit instead of one config.py change.

  Files added:
  - `templates/prompt_orchestrator.md`
  - `templates/prompt_workspace_agent.md`
  - `templates/prompt_focused_agent.md`

- **`bot/scheduler.py`** — the one-shot/periodic dispatcher and the
  continuous-task dispatcher both had a 17-line block that opened
  `output.log`, launched the CLI subprocess, piped the prompt to
  stdin, and handled the sync-vs-async StreamWriter quirk. Extracted
  into `_spawn_ai_subprocess()`. The lock-file write is now
  `_write_lock_file(path, pid)`. The two call sites are simpler and
  the orphan-safe ordering (state → lock) documented in v0.20.17
  becomes obvious because both paths reuse the same helper.

- **`bot/scheduler.py`** — `_reconcile_task_results()` now
  discriminates recurring vs one-shot dispatch by the presence of
  `interval_seconds` on the queue entry rather than by comparing the
  legacy `task_type == "periodic"` string. The behaviour is
  identical for every task that carries both fields (every task
  created by `add_task`), but the code is now harder to
  accidentally regress, and the `type` field becomes purely
  informational going forward.

### Skipped (and why)

- **`bot/handlers.py` single-pass pattern extraction** — the
  existing per-pattern blocks (`CREATE_WORKSPACE`,
  `CLOSE_WORKSPACE`, `CREATE_SPECIALIST`, `CREATE_CONTINUOUS`) are
  verbose but each one is self-documenting and locally greppable.
  Consolidating into a single `re.finditer` sweep would save about
  ten lines at the cost of introducing a cross-pattern ordering
  constraint. Not worth the risk for the marginal gain. The
  decision is recorded in the plan file.

- **`[REMIND ...]` double dispatch** — `_handle_remind_commands`
  is already a single regex + single attribute parser + single
  branch on the presence of `agent=`. The two downstream helpers
  (`_queue_text_reminder` / `_queue_action_reminder`) write to
  different queues because they mean different things (plain-text
  delivery vs scheduled agent invocation). Docstring expanded to
  make the contract explicit, no functional change.

### Tests

971 passed, 1 skipped, 3 warnings. Every existing test exercises
the refactored spawn path or the prompt-loading path indirectly.

### No migration needed

The extracted prompts are loaded at import time from the
repo-bundled `templates/` directory, so no on-disk state changes.
`v0_20_18.py` is a no-op required only for chain continuity.
