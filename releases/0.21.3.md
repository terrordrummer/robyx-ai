## Fixed

### Security — image decompression-bomb defence

- **P2-50 — `bot/media.py` rejects pathological images before Pillow
  decodes them.** Pillow's default behaviour for suspiciously-large
  images is to emit a `DecompressionBombWarning` (a Python warning, not
  an exception) and then go ahead and allocate the pixmap. No
  pre-Pillow file-size cap existed either, so a 10 GB crafted PNG
  would reach the decoder before any check kicked in.

  Three layered defences added:

  1. **Pre-open file-size cap** — `_MAX_IMAGE_FILE_BYTES = 25 * 1024 * 1024`
     enforced via `src.stat().st_size`. Above the cap the function
     raises `MediaError` before `Image.open` is ever called, so a
     hostile path with a multi-GB image can never touch the decoder.
  2. **Lowered pixel ceiling** — `Image.MAX_IMAGE_PIXELS` is set at
     module import time to 50 MP (down from Pillow's default 89 MP).
     Generous enough for any consumer-camera photo, tight enough to
     short-circuit the "tiny compressed file, huge decoded pixmap"
     attack class. Pixels greater than `2 × MAX_IMAGE_PIXELS` raise
     `DecompressionBombError` automatically; pixels in `(MAX, 2 × MAX)`
     raise `DecompressionBombWarning`, which gate #3 promotes to error.
  3. **Warning promoted to error, scoped** — the `Image.open` / `load`
     block is wrapped in `warnings.catch_warnings()` with
     `simplefilter('error', Image.DecompressionBombWarning)` so any
     suspicious image in the warning zone now raises `MediaError`
     instead of silently allocating.

  Closes trust-boundary **TG-4** from
  `specs/002-full-code-review/trust-boundaries.md`.

### Natural interaction — i18n discipline

- **P2-01, P2-02, P2-03 — three hard-coded literals removed from
  `bot/handlers.py`.** Strings passed directly to `platform.reply()`
  bypassed the i18n layer:

  | Line | Before | After |
  |---|---|---|
  | 213 | `"Usage: /reset <name>"` | `STRINGS["reset_usage"]` |
  | 306 | `"Checking for pending update..."` | `STRINGS["update_checking_manual"]` |
  | 1350 | `"No users registered in this workspace."` | `STRINGS["collab_no_users"]` |

  Ripgrep confirms **zero remaining direct-literal violations** across
  the 105 messaging call-sites in the codebase. The ownership rule
  from `contracts/conversation-contract.md` §1 is now fully observed.

### Test infrastructure — regression safety nets

- **P2-60 — `tests/test_i18n_parity.py` locks in two properties.**

  1. `TestStringSubstitution` — parametrised over every key in
     `STRINGS` (≈75 keys, 2 tests each). For each key with `%s` /
     `%d` specifiers it instantiates with dummy args and asserts the
     result contains no leftover format markers. It also scans for
     stray `.format()`-style `{placeholder}` tokens that would signal
     a template-rendering bug (closes the residual risk flagged by
     Pass 1 F19). A future typo that replaces `%s` with `{name}` is
     caught at test time instead of leaking to users.

  2. `TestHelpParity` — asserts set-equality between the keys
     returned by `make_handlers()` and the `/command` tokens
     extracted from `help_text`, modulo a curated exclusion list
     (`start`, `help`, and internal dispatch keys like `message` /
     `voice`). A future command added without a help entry — or a
     help entry for a non-existent handler — becomes a build failure.

  Closes T097 (`/help` parity audit) and the implementation side of
  T101 (i18n substitution test).

## Re-scoped

- **T100** (locale parity) — the bot is single-locale (English).
  The parity dimension that actually matters today (`/help` ⟷
  handlers) is covered by `TestHelpParity`. A locale-parity test
  becomes relevant only if a second locale is added; at that point
  the existing harness is a 10-line addition.
