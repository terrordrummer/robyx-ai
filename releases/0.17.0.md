## Reminders that *do* work, not just announce it

v0.15 introduced the universal `[REMIND ...]` pattern as a cheap, Python-fired
"ping me at T" primitive. It was deliberately limited to plain text: the
reminder engine reads `data/reminders.json` once a minute and posts the
stored message verbatim. That covers "at 9am tell me to prepare the
meeting" but not "at 9am **run** the daily cleanup and report back" â€”
which has always required the heavier timed task queue and a Python
snippet the agent had to emit by hand.

v0.17 unifies the two. The `[REMIND ...]` pattern now has an optional
`agent="<name>"` attribute; when present, the reminder becomes an
**action** instead of a plain message. At the scheduled time the named
workspace or specialist agent is spawned as a one-shot run with `text=`
as its prompt. The user's natural-language path into the timed task
queue is now the same pattern they already use for reminders â€” no
Python snippets, no two grammars to remember.

```text
# Text mode (unchanged) â€” "at T *tell me* that"
[REMIND in="1h30m" text="meeting prep"]
[REMIND at="2026-04-09T09:00:00+02:00" text="đź“… dentist"]

# Action mode (new in v0.17) â€” "at T *do* that"
[REMIND at="2026-04-10T09:00:00+02:00" agent="cleanup" text="Run the daily cleanup and post the summary to your topic."]
[REMIND in="1h" agent="monitor" text="Snapshot the dashboard and report anomalies."]
```

The two modes are strictly disjoint: a text reminder never touches
`data/timed_queue.json`, an action reminder never touches
`data/reminders.json`. The handler routes each match to the correct
backing store. Mixing both modes in a single agent response works and
is tested.

### Action mode â€” routing and defaults

When `agent=` is present, `bot/handlers.py:_handle_remind_commands`:

1. Resolves the target via `AgentManager.get(name)`.
2. Rejects the entry inline if the target doesn't exist, or exists but is
   **not** a workspace/specialist (Kael itself is not a valid target â€”
   the coordination layer must not run scheduled work).
3. Defaults `thread_id` to the **target agent's own topic**, not the
   caller's, so the work's output lands in the agent's own thread.
   Explicit `thread="<id>"` still overrides.
4. Builds a timed-queue entry with:
   - `type: one-shot`
   - `agent_file: agents/<name>.md` or `specialists/<name>.md` (matching
     `agent_type`)
   - `prompt: <text value>`
   - `scheduled_at: <UTC ISO from parse_remind_when>`
   - `model: <target's stored model, else "balanced">`
   - `source: "remind"` â€” additive metadata tag so logs and future
     tooling can tell REMIND-originated tasks apart from manually
     queued ones. **Ignored by every existing consumer**; backward-compat
     verified by a find_due() probe on mixed old/new entries.
5. Appends via `timed_scheduler.add_task(...)` â€” the same atomic
   write-then-rename path used by every other caller.

At fire time the existing `timed_scheduler.run_timed_cycle` dispatches
the entry like any other one-shot task. No new scheduler code path â€”
the action reminder is just a new producer for an existing queue.

### Validation and error handling

All validation runs before any file I/O, same discipline as the
pre-existing text-mode path:

| Error condition | Result |
|---|---|
| `text=` missing or empty | inline error, nothing queued |
| `in=` / `at=` both present or both absent | inline error, nothing queued |
| Invalid duration or naive `at=` | inline error, nothing queued |
| `agent=` names unknown agent | inline error, **neither queue touched** |
| `agent=` names Kael / orchestrator | inline error, type-guard rejection |
| `thread=` not an integer | inline error, nothing queued |

Inline errors are appended to the user-visible reply (same convention as
SEND_IMAGE validation), never silently dropped.

`WORKSPACE_AGENT_SYSTEM_PROMPT`, `FOCUSED_AGENT_SYSTEM_PROMPT` in
`bot/config.py`) and the human-facing `ORCHESTRATOR.md` now document:

- The full `dhms` compact-duration grammar for `in=` (days â†’ hours â†’
  minutes â†’ seconds, strict order, positive, â‰¤ 90 days).
- The ISO-8601 rules for `at=` (timezone offset required, 60s clock

## Reserved / duplicate agent names (M3)

`topics.create_workspace` and `topics.create_specialist` now raise
`ValueError` **before any side effects** when the sanitised name is:

1. In the new `RESERVED_AGENT_NAMES` frozenset (`kael`, `orchestrator`,
   or the empty string â€” the last is what `_sanitize_task_name` returns
   for inputs made entirely of punctuation).
2. Already taken by another registered agent (workspaces and
   specialists share a single `AgentManager` namespace â€” the earlier
   code would cheerfully overwrite the live entry in `manager.agents`
   and brick Kael).

`bot/handlers.py` catches the rejection distinctly from a generic
failure and surfaces the reason verbatim to the user ("Workspace *kael*
not created: cannot create workspace 'kael': name is reserved") instead
of the previous generic "Failed to create workspace *kael*". The guard
runs before `platform.create_channel` is called, so rejected
creations leave **no** partial state behind â€” no orphan topic, no
half-written `tasks.md` row, no agents/<name>.md file.

Six new parametrized tests in `tests/test_topics.py::TestReservedAndDuplicateNames`
cover every reserved spelling (`kael`, `Kael`, `KAEL`, `orchestrator`),
the empty-sanitised-name case (`!!!`), duplicate workspace names with
matching sanitisation, duplicate specialist names, and the
workspace â†” specialist cross-namespace collision.

## `models.yaml` fallback visibility (L4)

Before v0.17 the model-preference loader in `bot/config.py` silently
fell through three layers (`models.yaml` â†’ `AI_MODEL_*` env JSON â†’
hard-coded defaults) without telling anyone which layer it actually
used. A fresh clone with no `models.yaml` would quietly bill the
hard-coded tier; a setup with a typo in the yaml file would silently
fall back to env vars. Both scenarios were invisible until someone
grepped the resolved model id at invocation time.

v0.17 extracts the decision into a standalone
`bot/config.py:_log_models_fallback_source(...)` and calls it once at
module import. The function emits one `INFO` line covering every
branch:

```
Model preferences loaded from /â€¦/models.yaml
models.yaml not found at /â€¦/models.yaml â€” falling back to AI_MODEL_* env vars
models.yaml not found at /â€¦/models.yaml â€” falling back to hardcoded defaults (orchestrator=balanced, workspace=balanced, specialist=powerful)
models.yaml empty or unparseable at /â€¦/models.yaml â€” falling back to â€¦
models.yaml PyYAML not installed at /â€¦/models.yaml â€” falling back to â€¦
```

The hard-coded fallback explicitly names each role's default so an
operator scanning `bot.log` immediately sees which tier they are being
billed for. Four new tests in
`tests/test_model_preferences.py::TestModelsYamlFallbackLogging` lock
in every branch of the decision tree.

## Compatibility

- **Non-breaking for users.** Every existing `[REMIND ...]` pattern
  without `agent=` continues to behave exactly as in v0.16 (plain text
  reminder via `data/reminders.json`). The action mode is purely
  additive â€” it activates only when the agent explicitly sets `agent=`.
- **Non-breaking for `data/timed_queue.json`.** The new `source:
  "remind"` field on REMIND-originated entries is additive metadata;
  `timed_scheduler.find_due` and `dispatch_task` ignore unknown fields.
  Mixed old/new queues dispatch uniformly â€” verified by direct
  find_due() probe.
- **Non-breaking for the `[CREATE_WORKSPACE]` / `[CREATE_SPECIALIST]`
  emitters.** The reserved-name guard only rejects names that would
  have corrupted the `AgentManager` state (overwriting Kael,
  registering a nameless agent). Any name a previous version would have
  accepted *and* that wouldn't have broken the bot continues to be
  accepted.
- **No schema migrations.** `reminders.json`, `timed_queue.json`,
  `state.json`, `tasks.md`, `specialists.md`, and
  `agents/<name>.md` are all unchanged in shape.
- **No new dependencies.**
- **Sessions will be reset on first boot post-upgrade** because
  `bot/config.py` is in `session_lifecycle.GLOBAL_INVALIDATION_FILES`
  and the system prompts changed (the Reminders section in all three
  prompts now documents `agent=` action mode). This is the v0.15.1
  diff-driven invalidation doing exactly what it was designed to do â€”
  the change is picked up on the next auto-update cycle with zero
  manual action, and the next message to each agent creates a fresh
  Claude session that bakes in the v0.17 prompt. `manager.reset_sessions`
  handles the in-memory update; no direct `state.json` mutation.
