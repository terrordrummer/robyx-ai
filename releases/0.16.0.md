## Repo cleanup: runtime data belongs in `data/`

v0.16 finishes the cleanup that v0.15.2 started. Before this release,
KaelOps shipped **personal runtime data** committed alongside the
source code:

```
tasks.md                    ← the user's scheduled tasks, with real thread IDs
specialists.md              ← the user's specialists, with real thread IDs
agents/assistant.md         ← the user's assistant brief
agents/assistant-check.md   ← the user's secretary brief
agents/kael-ops-project.md  ← the user's self-management agent
agents/system-monitor.md    ← the user's scheduled task agent file
specialists/deploy.md       ← the user's deploy specialist
```

That was wrong for three reasons:

1. **Fresh clones were not pristine.** A fresh clone of the repo pulled
   rows pointing at agent files that existed only in the author's
   working directory. The scheduler errored on every cycle with
   `Agent file not found`. v0.15.2 was a surgical fix for exactly one
   such leak (the `zeus-engine` row) but the underlying architectural
   problem was still there.
2. **The author's personal setup was published on GitHub.** Thread IDs,
   agent names, project paths, personality text — all of it in the
   repo. A privacy and coupling problem.
3. **Auto-update was fragile.** Any edit the user made to their live
   `tasks.md` (e.g. Kael creating a new workspace) diverged from the
   tracked version, and the next auto-update either merge-conflicted or
   silently overwrote the user's state.

v0.16 is the architectural fix: **runtime data now lives under
`data/`** (already gitignored), and the repo root ships a clean shell
with zero personal entries.

## The move

```
tasks.md            → data/tasks.md
specialists.md      → data/specialists.md
agents/<name>.md    → data/agents/<name>.md
specialists/<n>.md  → data/specialists/<n>.md
```

`bot/config.py` path constants `TASKS_FILE`, `SPECIALISTS_FILE`,
`AGENTS_DIR`, `SPECIALISTS_DIR`, and `STATE_FILE` are all now derived
from `DATA_DIR`. The in-table agent column still reads
`agents/<name>.md`; `scheduler.spawn_task` and
`timed_scheduler.dispatch_task` resolve it against `DATA_DIR` rather
than `PROJECT_ROOT`, so existing `tasks.md` content is valid verbatim —
no row rewrite needed.

## The migration: how the Mac runtime install survives the pull

This is the critical part. The auto-updater does
`git stash --include-untracked → git pull --ff-only → stash pop`. If
v0.16 only does `git rm --cached tasks.md agents/...`, the pull will
**delete those files from the working tree** on the next auto-update
cycle — and the user's personal fleet state disappears.

The fix is a pre-pull migration inside `apply_update`:

```
apply_update(version, notify_fn, manager):
    1. git stash --include-untracked
    2. git rev-parse HEAD                      # pre-pull SHA (v0.15.1+)
    3. migrate_personal_data_to_data_dir()     # NEW in v0.16, PRE-pull
    4. git pull --ff-only
    5. pip install
    ...
    7. stash pop
    8. session invalidation via manager        # v0.15.2+
    9. restart_service()
```

`migrate_personal_data_to_data_dir()` copies every tracked runtime
file into `data/` **before** the pull removes the source copies from
the working tree. It is idempotent: files that already exist under
`data/` are never overwritten, so re-running on subsequent updates (or
downgrading and re-upgrading) does not clobber a newer `data/` copy.

On the user's Mac runtime install, the call copies:

- `tasks.md → data/tasks.md`
- `specialists.md → data/specialists.md`
- every `agents/*.md` (tracked AND untracked — the untracked ones like
  `agents/zeus-engine.md` get caught too)
- every `specialists/*.md`

Then the pull runs, removes the tracked repo-root copies from the
working tree, and the bot boots reading from `data/`. **Zero manual
intervention. Zero data loss.**

### The bootstrap safety net

`bot/_bootstrap.py` gains an analogous `migrate_personal_data_if_needed()`
that runs at every boot, before any bot-local imports. This covers the
alternative path where the user manually runs
`git pull && systemctl restart kaelops` without going through
`apply_update`. On that path, the pull has already removed the tracked
files — but any **untracked** leftovers (manually created briefs like
`agents/zeus-engine.md`) are still in the working tree and need to be
relocated. The bootstrap migration scoops them up on the next boot.

Same idempotency guarantee. Uses only the stdlib because it runs
before any third-party imports, just like the existing dependency
check.

## Fresh-clone behaviour

A fresh `git clone` of kael-ops now has:

```
kael-ops/
├── bot/...
├── install/...
├── releases/...
├── tests/...
├── README.md, CHANGELOG.md, VERSION, ...
└── (no tasks.md, no specialists.md, no agents/, no specialists/)
```

On first boot:

1. `bot/_bootstrap.py` runs the safety-net migration → no-op (no source
   files).
2. `bot/config.py` computes `TASKS_FILE = data/tasks.md` etc.
3. `AgentManager()` loads `data/state.json` → doesn't exist → empty
   fleet with only `kael`.
4. Kael is online on the Headquarters. No scheduled tasks. No
   specialists.
5. The user creates workspaces/specialists via Kael. `bot/topics.py`
   creates `data/agents/` / `data/specialists/` on the fly as needed.

No more "Agent file not found" errors for phantom agents that only
existed in the author's working directory.

## Compatibility

- **Non-breaking for users.** The pre-pull migration runs before the
  pull removes the source files, and the boot-time safety net catches
  any leftover files on the next restart. No manual action required.
- **Non-breaking for the `tasks.md` format.** Rows keep the existing
  `agents/<name>.md` column value; the resolver just points at
  `data/agents/<name>.md` instead of `agents/<name>.md`. Legacy rows
  migrate with the file.
- **Sessions will be reset** on first boot post-upgrade because
  `bot/config.py` (a `GLOBAL_INVALIDATION_FILES` entry) changes. This
  is expected: the diff-driven invalidation from v0.15.2 triggers, the
  manager-aware reset runs, and every agent gets a fresh AI-CLI
  session. The pre-pull personal-data migration is ordered before the
  pull, so the reset does not lose any fleet state — only the session
  IDs are regenerated.
- **Working-tree leftovers are not auto-cleaned.** After the upgrade,
  the old `agents/` and `specialists/` directories may still exist on
  the runtime install because they contained untracked files. These
  are now unreferenced; `rm -rf agents specialists` on the runtime
  install is safe once you have confirmed `data/agents/` and
  `data/specialists/` contain what you expect. A future release may
  add an automatic cleanup step.
- **No new dependencies.**
- **No `requires_migration: true`** — the migration is in-process,
  not a shell script.
