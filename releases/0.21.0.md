# Release 0.21.0

**SQLite-backed memory engine + second-pass security hardening**

min_compatible: 0.20.0
breaking: false

## Summary

This release rolls three streams of work into one version:

1. **SQLite memory engine** — the per-agent memory system moves from
   plain markdown files (`active.md` + quarterly archive directories)
   to per-agent SQLite databases with FTS5 full-text search. WAL mode
   enabled, automatic migration at first boot.
2. **Full code-review Pass 1** — a module-by-module review closed
   15 bug fixes, 3 security fixes, and removed dead code across 29
   modules.
3. **Code-review Pass 2 — security hardening** — a second pass
   re-read the same modules through four new lenses (Security,
   Stability, Ease of use, Natural Interaction). Two High-severity
   findings shipped as fixes in this release; the rest of Pass 2
   is in flight on branch `002-full-code-review`.

Behaviour for the common case is unchanged. What changes is **where
agent memory lives** (transparent migration) and **how the bot
protects itself from concurrent-start races and Slack-side redirect
abuse**.

## Memory engine (v0.21.0 headline feature)

Before 0.21.0 every agent had a `data/memory/<name>/active.md` file
plus an `archive/` directory of dated fragments. The new engine stores
the same information in `data/memory/<name>.db` with two tables:

- `active_snapshots` — one row per agent, the compact context payload.
- `entries` — append-only archive, indexed by `entries_fts` (FTS5).

Why:

- **Crash safety.** WAL mode means a mid-write crash leaves the
  database in a replay-consistent state. The old markdown files had
  no atomicity guarantee.
- **Archive search.** BM25-ranked full-text queries with snippet
  highlighting, instead of grepping through dated fragments.
- **Concurrent readers.** A long-running archive query no longer
  blocks an active-snapshot write.
- **Schema discipline.** Explicit tables with `created_at` /
  `archived_at` / `archive_reason` columns replace ad-hoc markdown
  conventions.

Integration with existing projects is unchanged — Robyx still respects
native Claude Code memory (`CLAUDE.md`, `.claude/` directories) and
does **not** create a `.db` for workspace agents whose projects have
their own memory.

See [`docs/memory.md`](../docs/memory.md) for the updated contract
and [`docs/data-directory.md`](../docs/data-directory.md) for the new
`data/memory/*.db` file table.

## Code review — Pass 2 security fixes

Pass 2 reframes the review around four lenses (Security, Stability,
Ease of Use, Natural Interaction) and produces foundational inventories
(threat model per adapter, crash-survival matrix, user-visible string
inventory). Two High-severity findings were fixed ahead of the full
Pass 2 traversal because their blast radius is small and the fixes
are self-contained:

### P2-10 — Slack bot token exfiltration via redirect (High)

**`bot/messaging/slack.py`**

`SlackPlatform.download_voice()` fetched the `file_id` URL with
`httpx.AsyncClient(follow_redirects=True)` and a `Authorization: Bearer
<bot_token>` header. There was no host allow-list on `file_id`, and
httpx replays Authorization verbatim across redirects. A crafted Slack
event (or a 302 `Location` pointing anywhere) could exfiltrate the
Slack bot token to an attacker-controlled host.

Fix:

- New `_validate_slack_file_url()` guard. Accepts HTTPS URLs on
  `files.slack.com`, `files.slack-edge.com`, or `slack-files.com`
  (and subdomains thereof); rejects everything else.
- `download_voice()` now calls the guard before the request and
  disables automatic redirect following. Redirects are followed
  manually, up to 5 hops, re-validating every `Location` header
  before the bearer token is replayed.

Tests: `tests/test_slack_platform.py` gets 10 new cases — four
`download_voice` scenarios (non-Slack host / non-HTTPS /
cross-host redirect / in-allow-list redirect) and six unit tests on
the URL guard (HTTPS enforcement, subdomain acceptance,
lookalike-host rejection, empty-host rejection).

### P2-20 — Single-instance PID-lock TOCTOU race (High)

**`bot/bot.py`**

`ensure_single_instance()` used a classic time-of-check /
time-of-use pattern:

```python
if PID_FILE.exists():
    pid = int(PID_FILE.read_text().strip())
    if is_pid_alive(pid): sys.exit("already running")
PID_FILE.write_text(str(os.getpid()))
```

Two processes starting within the race window could both pass the
check and both write their own PID, silently breaking the
single-instance invariant. Meanwhile `scheduler.py:98` and
`collaborative.py:153` already used `fcntl.flock` for their
cross-process locks — the correct pattern was in the codebase, just
not used here.

Fix:

- Replaced with a POSIX `fcntl.LOCK_EX | LOCK_NB` advisory lock on a
  sidecar file `data/bot.pid.lock`. The fd is held in a module-level
  global for the life of the process; the kernel releases the lock
  automatically on exit (even on SIGKILL), so a crashed owner never
  keeps the lock stuck.
- On platforms without `fcntl` (Windows), the legacy PID-file check
  is preserved as a fallback.
- `data/bot.pid` is still written (for introspection by keep-alive
  scripts) but is no longer load-bearing for mutual exclusion.

Tests: `tests/test_bot.py` rewrites `TestEnsureSingleInstance`
around the new fcntl path — lock-held exit (with and without PID
file), lock-fd retention for process lifetime, Windows fallback via
`__import__` patching. A `_reset_pid_lock` fixture prevents fd
leakage across tests.

## Migration

`bot/migrations/v0_21_0.py` handles the markdown-to-SQLite memory
migration. It is idempotent: a second run is a no-op, and a fresh
install (no existing markdown) is also a no-op.

The Pass 2 security fixes require no migration — they're code-only
changes. `data/bot.pid.lock` is created on first boot; no user
intervention needed.
