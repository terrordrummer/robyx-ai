## Workspace agents learn continuous tasks + updater ls-remote + README overhaul

Three independent improvements, all non-breaking, no schema changes.

### Added — workspace agents know about continuous tasks

Previously, only the orchestrator (`Robyx`) was documented to emit
`[CREATE_CONTINUOUS …]`. A workspace agent receiving a request for
iterative long-running work (research loop, optimization cycles,
training/evaluation iterations) had no mention of the continuous-task
mechanism in its system prompt, so it would silently start executing the
work in its own chat — flooding the topic and bypassing the dedicated
🔄 channel entirely.

The handler layer (`bot/handlers.py:511-557`) already accepts
`CREATE_CONTINUOUS` from any agent and attributes `parent_workspace` via
`manager.get_by_thread(thread_id)`, so no code change was needed — only
the workspace agent prompt template was missing the instructions.

**`templates/prompt_workspace_agent.md`** now has a dedicated
"Continuous Tasks" section covering:

- When to trigger the flow (research loops, optimization, iterative work
  with "repeat → measure → decide → repeat" structure).
- The setup interview (objective, success criteria, constraints,
  checkpoint policy, first step, context).
- The `[CREATE_CONTINUOUS name="…" work_dir="…"]` +
  `[CONTINUOUS_PROGRAM]{…}[/CONTINUOUS_PROGRAM]` emission pattern, with
  `work_dir` = the workspace's own path.
- The hand-off: after creation all interaction moves to the 🔄 topic.

Rationale for keeping the skill duplicated in workspace agents (instead
of escalating to the orchestrator): the workspace agent owns the project
context. The orchestrator doesn't know the project's domain, so routing
the interview there would force the user to re-explain the problem.
Keeping the interview where the knowledge is keeps conversations natural.

### Changed (performance) — `bot/updater.py`

- **`fetch_remote_tags()`** now uses `git ls-remote --tags --refs origin
  v*` instead of `git fetch --tags --force`. Previously the update check
  pulled every tag object from origin on every tick; with the release
  count growing monotonically, the fetch cost crept up. `ls-remote` is a
  lightweight ref lookup — bandwidth scales with the number of tag
  names, not tag objects.
- **`_get_release_notes_for()`** now lazy-fetches just the single tag
  needed when reading release notes (previously all tags were already
  local after `fetch --tags`). Falls back gracefully if the fetch fails.
- **`tests/test_updater.py`** updated to mock the new `ls-remote` flow.

### Changed (docs) — README overhaul

`README.md` previously jumped from "Why Robyx" (philosophy) straight to
the Quick Start. Readers had to click through the docs table to
understand *what Robyx actually does* and *how orchestration works*.

Two new sections inserted before the Documentation table:

- **Core Concepts** — compact tables for the three roles (Robyx,
  workspaces, specialists) and the four task types (interactive,
  one-shot, scheduled/periodic, continuous).
- **How Orchestration Works in Practice** — a seven-step narrative of a
  typical user journey: ask Robyx → workspace created on the fly → talk
  to the new agent → delegate/focus → continuous-task spin-up →
  reminders/timers → persistence across restarts.
- **Main Features** — single-glance bulleted list of the concrete
  capabilities (platforms, backends, scheduler, continuous tasks,
  interruption, memory, voice/images, auto-updates, permissions,
  service).

The Documentation table is unchanged and still links to the detail
pages under `docs/`.
