## Timed Task Queue — Precise Scheduling

Introduces a two-tier scheduling architecture that fixes missed one-shot tasks
and enables any agent to programmatically schedule work at a precise future time.

### Problem solved

The previous single-loop scheduler ran every 10 minutes. A one-shot task
scheduled at 12:00 could be dispatched at 12:00, 12:10, or anywhere in between
depending on the polling cycle — and if the dispatch window straddled the hour
boundary the task would be seen as past and silently dropped.

### Architecture

**Tier 1 — Periodic scheduler** (unchanged): `tasks.md` + 10-minute loop.
Handles infrastructure agents that repeat on fixed intervals (hourly, daily, etc.).

**Tier 2 — Timed queue** (new): `data/timed_queue.json` + 60-second loop.
Handles one-shot tasks and dynamic periodic tasks created at runtime by agents.

### New: `data/timed_queue.json`

Runtime queue for timed tasks. Schema per entry:

```json
{
  "id": "<uuid>",
  "name": "remind-meeting",
  "agent_file": "agents/my-workspace.md",
  "prompt": "Tell Roberto: meeting in 30 minutes!",
  "type": "one-shot",
  "scheduled_at": "2026-04-10T15:00:00+00:00",
  "status": "pending",
  "model": "claude-haiku-4-5-20251001",
  "thread_id": "12345",
  "created_at": "2026-04-08T10:00:00+00:00"
}
```

For `type: "periodic"` also include `"interval_seconds": <n>`.

### Jitter / offline recovery

Tasks whose `scheduled_at` or `next_run` is in the past are dispatched
immediately on the next cycle. No event is ever lost due to a restart or
brief outage.

### Race-condition safety

All queue mutations use atomic `write-then-rename` (`os.replace`), which is
POSIX-guaranteed atomic. The queue file is never in a partially-written state.

### Automatic migration

On startup, any one-shot rows still in `tasks.md` are automatically moved to
`data/timed_queue.json` and disabled in `tasks.md`. No manual action required.

### Agents can schedule tasks

Every workspace agent now has the queue API documented in its system prompt.
A running agent can append to `data/timed_queue.json` to schedule a future
callback — reminders, follow-ups, multi-step sequences with delays.

### New `.env` key

`TIMED_SCHEDULER_INTERVAL` — polling interval in seconds (default: `60`).

## Compatibility

No breaking changes. The `tasks.md` format is unchanged for `scheduled` and
`interactive` tasks. The timed queue file is created on first use. Safe upgrade
from v0.12.x; one-shot tasks in `tasks.md` are migrated automatically at boot.
