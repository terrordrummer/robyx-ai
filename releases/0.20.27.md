# Release 0.20.27

**Silent-by-default scheduled delivery + collaborative workspace hardening**

min_compatible: 0.20.0
breaking: false

## Summary

Two independent fixes shipped together.

1. **Silent-by-default for scheduled task delivery.** Periodic and
   continuous agents no longer flood Telegram with "all clear" status
   reports (the notorious "Assistente personale check" and "System
   Monitor" tables). Agents are now instructed to answer `[SILENT]`
   when a run has nothing actionable to report, and the delivery layer
   suppresses the Telegram post. Failures are never silent.

2. **Collaborative workspaces hardening.** Code review of v0.20.26
   surfaced several correctness and security issues in the new collab
   feature. This release addresses all critical/high findings plus the
   medium/low cleanup pass.

## Scheduled delivery

- `bot/scheduled_delivery.py` detects `[SILENT]` on a zero-returncode
  run and skips `send_to_channel` entirely. On failure the marker is
  stripped and the normal error message is rendered.
- `bot/scheduler.py` wraps every one-shot/periodic prompt with an
  explicit OUTPUT POLICY that tells the agent to answer `[SILENT]`
  when there is no anomaly, deadline, or decision to surface.
- `templates/CONTINUOUS_STEP.md` carries the same policy for
  continuous-step agents. Logs, state updates, and commits still
  happen; only the chat delivery is muted.

## Collaborative workspaces

### Security & correctness

- **Fail-closed owner check.** `authorization.get_user_role` accepts
  `owner_id=None` and no longer falls back to `0`, so an unset
  `OWNER_ID` never grants anyone the OWNER role.
- **Pending workspaces bound to a creator.** `CollabWorkspace` gained
  an optional `expected_creator_id` and `CollabStore` exposes
  `list_pending_for_creator()`. Flow A now matches a pending workspace
  only against the user who added the bot â€” an outsider can no longer
  hijack a workspace provisioned for someone else.
- **Setup workspaces are routable.** `_rebuild_chat_map` includes
  `status="setup"`, unblocking Flow B (bot added to an unknown group â†’
  in-group setup) whose first reply previously fell into the HQ
  owner-only path.
- **Agent never mutates roles.** Unknown senders now default to
  `PARTICIPANT` in-memory only. `/promote` and `/demote` (owner-only)
  remain the sole paths that change persisted roles. Telegram group
  membership is the authorization signal for PARTICIPANT.
- **Defense-in-depth against prompt injection.** `_process_and_send`
  takes an `is_executive` flag; non-executive responses run through
  `_strip_executive_markers`, which drops every tool-marker
  (`FOCUS`, `RESTART`, `CREATE_WORKSPACE`, `REMIND`, `SEND_IMAGE`, â€¦).
  PARTICIPANT prompt-injection attempts can no longer trigger system
  actions even if the LLM obeys them.
- **Concurrent-safe store.** `CollabStore` serialises every mutation
  through a threading-lock + `fcntl.flock` mutex, matching the
  scheduler's queue-mutex pattern.

### Cleanup

- Removed all callsites that reached into `CollabStore._workspaces` or
  `_save()`. Public `list_all()` / `list_pending_for_creator()` /
  `add()` / `update_roles()` cover every use case.
- `Platform.bot_username` is now a public property on the platform
  adapter; the telegram adapter returns `bot.username`. Passive-mode
  mention detection no longer reaches for a private attribute.
- `can_close_workspace` accepts an optional `owner_id` and is used by
  the `/close` handler â€” previously the rule was inlined with a
  different (but equivalent) check.
- `_collab_role` tolerates unknown role strings in a hand-edited
  store, falling back to `PARTICIPANT` with a warning instead of
  crashing Flow A.
- `CollabStore.purge_closed()` gives operators a way to drop stale
  closed workspaces without deleting the JSON file by hand.
- Welcome message in Flow A now mentions owner *and* operators as
  executive roles.

## Migration

No data migration required. `expected_creator_id` is optional on load,
so existing `collaborative_workspaces.json` files parse unchanged.
