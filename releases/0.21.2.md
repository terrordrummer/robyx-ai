## Fixed

### Stability — migration tracker durability

- **P2-40 — `bot/migrations/tracker.py` `save()` is now atomic.**
  The previous implementation wrote `migrations.json` with
  `path.write_text(json.dumps(...))` — no temp file, no `fsync`, no
  atomic rename. A SIGKILL, `kill -9`, or host power loss in the
  narrow window between opening the file and finishing the write
  could leave the file half-serialised.

  On the next boot, `load()` catches the `JSONDecodeError` and
  treats the tracker as empty. The migration runner then treats the
  state as "never migrated" and re-runs **every** migration in the
  chain. That is safe only if every migration step is strictly
  idempotent. Any migration that assumes it already ran (e.g. the
  markdown-to-SQLite conversion in `v0_21_0`, which renames the
  source files after conversion) could misbehave on the second run.

  Rewritten to go through `tmp + fsync + os.replace`. `fsync`
  failures are swallowed to tolerate filesystems that don't support
  it (tmpfs, some network mounts) but the rename itself is always
  atomic. A regression test simulates a failing `os.replace` and
  verifies the original file is byte-identical afterward — a
  non-atomic `write_text()` would have already truncated it.

  Closes **C2** from `specs/002-full-code-review/crash-matrix.md`.

## Re-scoped (noted for transparency)

The Pass 2 inventories on branch `002-full-code-review`
pre-identified six gaps across `trust-boundaries.md` and
`crash-matrix.md`. After reading the code, two items turned out to
not be what the inventory described:

### T074 — `.env` hot-reload mutex → no-action

The trust-boundary X-3 concern assumed a `.env` hot-reload mechanism
existed in the codebase and needed a mutex to serialise with
in-flight AI invocations. There is **no hot-reload**:
`bot/config_updates.py` is a writer only, `.env` is read once at
startup via `load_dotenv()` in `bot/config.py`, and the user-facing
i18n copy (`voice_no_key`, etc.) explicitly instructs users to
restart the bot after editing `.env`. Finding retracted.

### T080 — scheduler monotonic clock → deferred to its own spec

Crash-matrix C6 flagged wall-clock usage in `bot/scheduler.py` as a
stability risk (NTP jumps could double-fire or skip retries). On
audit, the scheduler uses `datetime.now(timezone.utc)` (not
`time.time()`), and the overwhelming majority of those uses are
legitimately wall-clock: serialised timestamps (`created_at`,
`sent_at`, `canceled_at`), absolute deadlines (`fire_at`,
`scheduled_at`), and recurring-task advancement (`_next_run_after`).

A correct fix would require a design that separates *intervals*
(stale-claim TTL, reminder age, retry backoff — `time.monotonic`
is right) from *deadlines* and *serialisation* (wall-clock is
right), and threads the interval references through in-memory
state that can survive across ticks but not across restarts. That
is spec-scale work, not one-slice work. **Deferred with rationale**
recorded as finding P2-80. A future feature branch can pick it up.
